## 什麼是 Agentic Loop

傳統聊天機器人：你問一句，它答一句。  
Claude Code 不一樣：你說一個需求，它可能連續執行十幾步操作纔給你最終結果。

這背後的機制叫做 **Agentic Loop**（智能體循環），核心實現在 `src/query.ts` 的 `queryLoop()` 異步生成器函數（第 241 行）。它是一個 `while(true)` 無限循環，每次迭代代表一次"思考→行動→觀察"週期。

## 循環的完整結構

`queryLoop()` 的每次迭代（`src/query.ts:307` `while(true)`）包含以下階段：

### 階段 1：上下文預處理（Pre-Processing Pipeline）

在呼叫 API 之前，依次執行 5 個壓縮/優化步驟：

```
messagesForQuery（原始訊息）
  ↓ applyToolResultBudget()    — 工具結果預算截斷（按 maxResultSizeChars）
  ↓ snipCompactIfNeeded()      — 歷史 Snip 壓縮（HISTORY_SNIP feature）
  ↓ microcompact()             — 微壓縮（工具結果摘要）
  ↓ applyCollapsesIfNeeded()   — 上下文摺疊（CONTEXT_COLLAPSE feature）
  ↓ autocompact()              — 自動壓縮（超出閾值時觸發）
messagesForQuery（處理後的訊息）→ 發往 API
```

每個步驟的輸出是下一步的輸入，形成串行管道。Snip 和 Microcompact 的釋放 token 數會傳遞給 autocompact 的閾值計算（`snipTokensFreed`），避免重複壓縮。

### 階段 2：流式 API 呼叫（Streaming Loop）

`deps.callModel()` 發起流式請求（第 659 行），返回一個 AsyncGenerator。在流式過程中：

- **AssistantMessage** 被收集到 `assistantMessages[]` 數組
- **tool_use 塊** 被提取到 `toolUseBlocks[]`，設置 `needsFollowUp = true`
- **StreamingToolExecutor** 在流式過程中就開始並行執行工具（不等流結束）
- 可恢復的錯誤（prompt-too-long、max-output-tokens）被**暫扣**（withheld），先嚐試恢復

流式回調中的關鍵守衛：
- `backfillObservableInput()`（第 763 行）—— 爲 tool_use 塊回填可觀察字段（如檔案路徑展開），但只在添加了新字段時才克隆訊息，避免破壞 prompt cache 的字節一致性
- 流式降級檢測——如果 `streamingFallbackOccured`，已收集的訊息被標記爲 tombstone（第 717 行），清空後重試

### 階段 3：工具執行（Tool Execution）

如果 `needsFollowUp` 爲 true，循環不會終止，而是執行工具：

```typescript
// 兩種工具執行器（互斥）
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()  // 流式：取得已完成的+等待中的
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)
```

工具結果通過 `normalizeMessagesForAPI()` 標準化後，與原始訊息合併，進入**下一輪循環迭代**。

### 階段 4：終止或繼續

每次迭代結束時，根據條件決定 `return`（終止）或 `continue`（繼續）：

## 7 種終止條件（源碼級）

| 終止原因 | 觸發位置 | 機制 |
|----------|---------|------|
| **completed** | 第 1360 行 | AI 未發出 tool_use → `needsFollowUp = false` → 經過 stop hooks → 返回 |
| **blocking_limit** | 第 646 行 | Token 計數超過硬限制（非 autocompact 模式）→ 生成 PTL 錯誤訊息 → 返回 |
| **aborted_streaming** | 第 1054 行 | `abortController.signal.aborted` → 爲未完成的 tool_use 生成合成 tool_result → 返回 |
| **model_error** | 第 999 行 | `callModel()` 拋出異常 → 生成錯誤訊息 → 返回 |
| **prompt_too_long** | 第 1178 行 | 413 錯誤且 reactive compact 無法恢復 → 暫扣的錯誤訊息被釋放 → 返回 |
| **image_error** | 第 980/1178 行 | 圖片尺寸/大小錯誤 → 直接返回 |
| **stop_hook_prevented** | 第 1282 行 | Stop hook 返回 `preventContinuation: true` → 返回 |

## 4 種繼續條件（恢復路徑）

循環不僅是一個簡單的"有 tool_use 就繼續"，它還包含多種恢復/重試路徑：

### 1. 正常工具循環
`needsFollowUp = true` → 執行工具 → 新訊息追加到 `messagesForQuery` → `continue`

### 2. max_output_tokens 恢復（第 1191-1255 行）
當 AI 輸出被截斷時（`apiError === 'max_output_tokens'`）：
- **首次**：嘗試將 `maxOutputTokens` 從預設值提升到 `ESCALATED_MAX_TOKENS`（64K），無 meta 訊息，靜默重試
- **後續**：注入恢復訊息"Output token limit hit. Resume directly..."，最多重試 `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3` 次
- 恢復耗盡後，暫扣的錯誤訊息被釋放

### 3. Prompt-Too-Long 恢復（第 1088-1186 行）
當遇到 413 錯誤時，有兩個恢復階段：
- **Context Collapse Drain**（第 1097 行）：提交所有已暫存的摺疊，釋放空間後重試。如果上一輪已經是 collapse_drain_retry 則跳過
- **Reactive Compact**（第 1123 行）：觸發即時壓縮，生成摘要後重試。`hasAttemptedReactiveCompact` 防止無限循環

### 4. Stop Hook 阻塞重試（第 1285-1308 行）
Stop hook 可以注入阻塞錯誤訊息，強制 AI 重新思考。新的訊息（包含阻塞錯誤）被追加到對話中，`stopHookActive = true`，進入下一輪迭代。

## 模型降級（Fallback）

當主模型不可用時（`FallbackTriggeredError`，第 897 行）：

1. 已收集的 `assistantMessages` 被清空，tool_use 塊收到合成 tool_result："Model fallback triggered"
2. 思維簽名塊被移除（`stripSignatureBlocks`）—— 因爲思維簽名與模型綁定，跨模型回放會 400
3. 切換到 `fallbackModel`，更新 `toolUseContext.options.mainLoopModel`
4. 生成系統訊息："Switched to {fallback} due to high demand for {original}"
5. 重新發起流式請求

## 狀態機：State 對象

每次迭代的狀態通過 `State` 類型（第 204 行）傳遞：

```typescript
type State = {
  messages: Message[]                        // 當前對話訊息
  toolUseContext: ToolUseContext              // 工具上下文（含權限）
  autoCompactTracking: AutoCompactTrackingState  // 壓縮跟蹤
  maxOutputTokensRecoveryCount: number       // 輸出截斷恢復計數
  hasAttemptedReactiveCompact: boolean       // 是否已嘗試即時壓縮
  maxOutputTokensOverride: number | undefined // 輸出 token 上限覆蓋
  pendingToolUseSummary: Promise<...> | undefined  // 異步工具摘要
  stopHookActive: boolean | undefined        // Stop hook 是否激活
  turnCount: number                          // 輪次計數
  transition: Continue | undefined           // 上一次繼續的原因
}
```

每次 `continue` 都建立新的 State 對象（不可變更新），而非就地修改。`transition` 字段記錄了爲什麼繼續——讓後續迭代能檢測特定恢復路徑（如 `collapse_drain_retry`）避免循環。

## Token Budget（實驗性）

當 `TOKEN_BUDGET` feature 啓用時（第 1311 行），循環在終止前會檢查 token 消耗：

- **continuation**：未達到預算但超過閾值 → 注入 nudge 訊息，讓 AI 加速收尾
- **diminishing_returns**：檢測到收益遞減 → 提前終止
- 預算資料來自 `createBudgetTracker()`，跨迭代累計

## 爲什麼不是"一次規劃，批量執行"

<Note>
源碼揭示了爲什麼 Claude Code 選擇逐步循環：
</Note>

- **每一步都產生真實信息**：`runTools()` 返回的 `toolResults` 是 API 不可能預知的——命令輸出、檔案內容、錯誤信息
- **動態上下文管理**：每輪迭代前都重新評估壓縮需求（autocompact → microcompact → snip），基於最新的 token 計數
- **錯誤即時恢復**：工具失敗不需要推倒重來——stop hook 可以注入阻塞錯誤讓 AI 修正策略
- **用戶可控**：`abortController.signal` 在循環的多個檢查點被檢測（第 1018、1048、1488 行），用戶按 ESC 可以優雅中斷
- **成本控制**：Token Budget 在每輪終止前檢查，防止 AI 無效循環

## 一個完整的迭代示例

> 用戶："幫我找到專案裏所有未使用的導入語句，然後刪掉它們"

```
迭代 1: 思考→行動
  預處理: 無需壓縮（上下文很短）
  API 呼叫: 返回 tool_use(Glob, "**/*.ts")
  工具執行: 返回 42 個檔案路徑
  → needsFollowUp = true, continue

迭代 2: 思考→行動
  預處理: 42 個文件結果仍在預算內
  API 呼叫: 返回 tool_use(Grep, "import.*from")
  工具執行: 在 15 個檔案中找到 120 條 import
  → needsFollowUp = true, continue

迭代 3: 思考→行動（多輪）
  預處理: 120 條 Grep 結果觸發 microcompact → 摘要化
  API 呼叫: 返回 3 個 tool_use(FileEdit, ...)
  工具執行: 刪除 5 條未使用導入
  → needsFollowUp = true, continue

迭代 4: 總結
  API 呼叫: 返回純文本"已清理 3 個檔案中的 5 條未使用導入"
  → needsFollowUp = false
  → Stop hooks 通過
  → return { reason: 'completed' }
```
