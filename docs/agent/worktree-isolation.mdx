## 爲什麼需要文件級隔離

多 Agent 並行工作時，共享同一工作目錄會導致三類衝突：

1. **寫入衝突**：兩個 Agent 同時編輯 `config.ts`，後寫的覆蓋前寫的
2. **狀態幹擾**：Agent A 的測試依賴某個環境狀態，Agent B 的修改破壞了它
3. **不可區分**：半完成的修改混在一起，無法分辨哪些是哪個 Agent 的

Git worktree 是 git 原生的解決方案——在同一個倉庫中建立多個獨立工作目錄，每個在自己的分支上。

## 目錄結構與命名規則

Worktree 文件統一存放在倉庫根目錄下的 `.claude/worktrees/`：

```
<repo-root>/
├── .claude/
│   └── worktrees/
│       ├── fix-auth-bug/          # worktree 工作目錄
│       │   ├── .git               # 指向主倉庫的連結文件
│       │   └── src/...            # 獨立的檔案系統視圖
│       └── add-dark-mode/         # 另一個 worktree
│           └── ...
├── src/                           # 主工作目錄（不受影響）
└── .git/                          # 主倉庫
```

分支命名規則爲 `worktree/<slug>`，其中 slug 由 `validateWorktreeSlug()` 校驗：每個 `/` 分隔的段只允許字母、數字、`.`、`_`、`-`，總長 ≤64 字符。未指定時使用 plan slug 自動生成。

## 建立流程：EnterWorktreeTool

`EnterWorktreeTool`（`src/tools/EnterWorktreeTool/EnterWorktreeTool.ts`）的執行鏈路：

```
EnterWorktreeTool.call({ name? })
  ↓
1. 檢查是否已在 worktree 中（防嵌套）
  ↓
2. 解析到主倉庫根目錄（findCanonicalGitRoot）
   如果當前已在 worktree 內，chdir 到主倉庫
  ↓
3. 生成 slug（用戶提供或 plan slug）
  ↓
4. createWorktreeForSession(sessionId, slug)
   ├── 有 WorktreeCreate hook？
   │   └── 執行 hook，返回 hook 指定的路徑（支援非 git VCS）
   └── 無 hook → git 原生路徑：
       a. getOrCreateWorktree(repoRoot, slug)
          ├── 快速恢復：檢查 worktree 目錄是否已存在
          │   └── 讀取 .git 指針文件的 HEAD SHA（無子進程）
          └── 新建：
              i.   mkdir .claude/worktrees/（recursive）
              ii.  fetch origin/<default-branch>（有快取則跳過）
              iii. git worktree add -b worktree/<slug> <path> <base>
              iv.  performPostCreationSetup()（sparse checkout 等）
  ↓
5. 更新進程狀態：
   - process.chdir(worktreePath)
   - setCwd(worktreePath)
   - setOriginalCwd(worktreePath)
   - saveWorktreeState(session) → 持久化到專案設定
   - clearSystemPromptSections() → 重新計算系統提示中的 cwd 信息
   - clearMemoryFileCaches() → 重新加載 worktree 中的 CLAUDE.md
  ↓
6. 返回 worktreePath 和 worktreeBranch
```

### Hook 優先的架構

`createWorktreeForSession()` 首先檢查 `hasWorktreeCreateHook()`——如果用戶在 settings.json 中設定了 `WorktreeCreate` hook，系統完全不呼叫 git，而是執行 hook 命令並將返回的路徑作爲 worktree 路徑。這允許非 git 版本控制系統（如 Pijul、Mercurial）通過 hook 接入。

### 快速恢復路徑

`getOrCreateWorktree()` 有一個關鍵優化：如果目標路徑已存在，直接讀取 `.git` 指針文件取得 HEAD SHA（純檔案 I/O，無子進程），跳過整個 `fetch` + `worktree add` 流程。在大倉庫中 `fetch` 需要 6-8 秒，這個優化將恢復場景的延遲降到接近 0。

## 退出流程：ExitWorktreeTool

`ExitWorktreeTool`（`src/tools/ExitWorktreeTool/ExitWorktreeTool.ts`）支援兩種退出策略：

### keep：保留 worktree

```
keepWorktree()
  ↓
1. chdir 回 originalCwd
2. 清空 currentWorktreeSession
3. 更新專案設定（activeWorktreeSession = undefined）
4. worktree 目錄和分支保留在磁盤上
```

用戶可以通過 `cd <worktreePath>` 繼續工作，或稍後手動合併。

### remove：刪除 worktree

有嚴格的**安全防護**：

```
validateInput() — 第一道防線
  ↓
1. 檢查是否在 EnterWorktree 建立的會話中
   （手動建立的 worktree 不會被刪除）
  ↓
2. countWorktreeChanges(worktreePath, originalHeadCommit)
   ├── git status --porcelain → 統計未提交文件數
   ├── git rev-list --count <originalHead>..HEAD → 統計新提交數
   └── 返回 null（git 失敗時）→ fail-closed（拒絕刪除）
  ↓
3. 有未提交文件或新提交？
   → 拒絕，要求 discard_changes: true 確認
```

```
call() — 實際執行
  ↓
1. 重新計數變更（validateInput 和 call 之間可能有新修改）
2. 如果有 tmux session → killTmuxSession()
3. cleanupWorktree()
   ├── hook-based → 執行 WorktreeRemove hook
   └── git-based → git worktree remove --force + git branch -D
4. restoreSessionToOriginalCwd()
   - setCwd(originalCwd)
   - setOriginalCwd(originalCwd)
   - 如果 projectRoot 是 worktree 時才恢復（防誤觸）
   - 更新 hooks config snapshot
   - 清空系統提示和 memory 快取
```

### fail-closed 設計

`countWorktreeChanges()` 在以下情況返回 `null`（"未知，假設不安全"）：
- `git status` 或 `git rev-list` 退出非零（鎖文件、損壞的索引）
- `originalHeadCommit` 未定義（hook-based worktree 沒有設置基線 commit）

返回 `null` 時，`validateInput` 拒絕刪除——寧可讓用戶手動處理，也不冒險丟失工作。

## 與 Agent 工具的聯動

Agent 工具（`AgentTool`）的 `isolation` 參數決定子 Agent 是否在 worktree 中執行：

- `isolation: "worktree"` → 呼叫 `createWorktreeForSession()`，子 Agent 在獨立 worktree 中執行
- 無 isolation → 子 Agent 共享主工作目錄

子 Agent 結束時的處理：
- **成功**：主 Agent 通過 `ExitWorktreeTool(action: "keep")` 保留 worktree，然後手動合併
- **失敗/放棄**：主 Agent 通過 `ExitWorktreeTool(action: "remove", discard_changes: true)` 清理

## Session 狀態持久化

`WorktreeSession` 對象通過 `saveCurrentProjectConfig()` 持久化到磁盤，包含：

```typescript
{
  originalCwd: string,         // 進入 worktree 前的工作目錄
  worktreePath: string,        // worktree 的絕對路徑
  worktreeName: string,        // slug
  worktreeBranch?: string,     // 分支名（如 worktree/fix-auth）
  originalBranch?: string,     // 進入前的分支
  originalHeadCommit?: string, // 進入前的 HEAD commit（用於變更統計）
  sessionId: string,           // 建立此 worktree 的會話 ID
  tmuxSessionName?: string,    // 關聯的 tmux session
  hookBased?: boolean,         // 是否由 hook 建立
  creationDurationMs?: number, // 建立耗時（分析用）
}
```

這使得 session 恢復（`--resume`）時能正確還原 worktree 上下文——即使進程重啓，`getCurrentWorktreeSession()` 從專案設定中讀取狀態。

## Sparse Checkout 優化

對於大型 monorepo，worktree 支援 `sparsePaths` 設定——只檢出特定目錄而非整個倉庫。這在 210K 文件的倉庫中將 worktree 建立時間從數十秒降到幾秒。

設定位於 `getInitialSettings().worktree?.sparsePaths`，在 `performPostCreationSetup()` 中應用。
