## 從數組到 API 呼叫：System Prompt 的完整鏈路

System Prompt 在 Claude Code 中不是一段寫死的文本，而是一個 **`string[]` 數組**（品牌類型 `SystemPrompt`，定義於 `src/utils/systemPromptType.ts:8`），經過組裝、分塊、快取標記後發送給 API。

### 三階段管道

```
getSystemPrompt()          →  string[]       （組裝內容）
  ↓
buildEffectiveSystemPrompt() →  SystemPrompt   （選擇優先級路徑）
  ↓
buildSystemPromptBlocks()  →  TextBlockParam[] （分塊 + cache_control 標記）
```

1. **`getSystemPrompt()`**（`src/constants/prompts.ts:444`）—— 收集靜態段 + 動態段，插入 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 分界標記
2. **`buildEffectiveSystemPrompt()`**（`src/utils/systemPrompt.ts:41`）—— 按 Override > Coordinator > Agent > Custom > Default 優先級選擇
3. **`buildSystemPromptBlocks()`**（`src/services/api/claude.ts:3214`）—— 呼叫 `splitSysPromptPrefix()` 分塊，爲每個塊附加 `cache_control`

## SystemPrompt 品牌類型

```typescript
// src/utils/systemPromptType.ts:8
export type SystemPrompt = readonly string[] & {
  readonly __brand: 'SystemPrompt'
}
export function asSystemPrompt(value: readonly string[]): SystemPrompt {
  return value as SystemPrompt  // 零開銷類型斷言
}
```

品牌類型（branded type）防止普通 `string[]` 被意外傳入 API 呼叫——只有通過 `asSystemPrompt()` 顯式轉換才能獲得 `SystemPrompt` 類型。

## getSystemPrompt()：內容組裝的全景

`src/constants/prompts.ts:444` 是 System Prompt 的核心工廠函數，返回一個有序數組：

| 階段 | 內容 | 快取策略 |
|------|------|----------|
| **靜態區** | Intro Section、System Rules、Doing Tasks、Actions、Using Tools、Tone & Style、Output Efficiency | 可跨組織快取（`scope: 'global'`） |
| **BOUNDARY** | `SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'` | 分界標記（不發送給 API） |
| **動態區** | Session Guidance、Memory、Model Override、Env Info、Language、Output Style、MCP Instructions、Scratchpad、FRC、Summarize Tool Results、Token Budget、Brief | 每次會話不同（`scope: 'org'` 或無快取） |

### 動態區的 Section 註冊表

動態區通過 `systemPromptSection()` / `DANGEROUS_uncachedSystemPromptSection()` 註冊，這兩個工廠函數定義於 `src/constants/systemPromptSections.ts`：

```typescript
// 快取式 Section：計算一次，/clear 或 /compact 後才重新計算
systemPromptSection('memory', () => loadMemoryPrompt())

// 危險：每輪重新計算，會破壞 Prompt Cache
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(mcpClients),
  'MCP servers connect/disconnect between turns'  // 必須給出破壞快取的理由
)
```

`resolveSystemPromptSections()` 在每輪查詢時解析所有 Section，對於 `cacheBreak: false` 的 Section，優先使用 `getSystemPromptSectionCache()` 中的快取值。只有 MCP 指令等真正動態的內容使用 `DANGEROUS_uncachedSystemPromptSection`。

### `CLAUDE_CODE_SIMPLE` 快速路徑

當環境變量 `CLAUDE_CODE_SIMPLE` 爲真時，整個 System Prompt 縮減爲一行：

```typescript
`You are Claude Code, Anthropic's official CLI for Claude.\n\nCWD: ${getCwd()}\nDate: ${getSessionStartDate()}`
```

跳過所有 Section 註冊、快取分塊、動態組裝——用於最小化 token 消耗的測試場景。

## buildEffectiveSystemPrompt()：五級優先級

`src/utils/systemPrompt.ts:41` 決定最終使用哪個 System Prompt：

| 優先級 | 條件 | 行爲 |
|--------|------|------|
| **0. Override** | `overrideSystemPrompt` 非空 | 完全替換，返回 `[override]` |
| **1. Coordinator** | `COORDINATOR_MODE` feature + 環境變量 | 使用協調者專用提示詞 |
| **2. Agent** | `mainThreadAgentDefinition` 存在 | Proactive 模式：追加到預設提示詞尾部；否則：替換預設提示詞 |
| **3. Custom** | `--system-prompt` 參數指定 | 替換預設提示詞 |
| **4. Default** | 無特殊條件 | 使用 `getSystemPrompt()` 完整輸出 |

`appendSystemPrompt` 始終追加到末尾（Override 除外）。

## 快取策略：分塊、標記、命中

這是 System Prompt 設計中最精密的部分。

### Anthropic Prompt Cache 基礎

Anthropic API 的 Prompt Cache 允許跨請求複用相同的 System Prompt 前綴，按快取命中量計費（遠低於完整輸入價格）。快取鍵由內容的 Blake2b 哈希決定——任何字符變化都會導致快取失效。

### `splitSysPromptPrefix()`：三種分塊模式

`src/utils/api.ts:321` 是快取策略的核心，根據條件選擇三種分塊模式：

#### 模式 1：MCP 工具存在時（`skipGlobalCacheForSystemPrompt=true`）

```
[attribution header]    → cacheScope: null     （不快取）
[system prompt prefix]  → cacheScope: 'org'    （組織級快取）
[everything else]       → cacheScope: 'org'    （組織級快取）
```

MCP 工具列表在會話中可能變化（連接/斷開），破壞了跨組織快取的基礎，因此降級爲組織級。

#### 模式 2：Global Cache + Boundary 存在（1P 專用）

```
[attribution header]    → cacheScope: null     （不快取）
[system prompt prefix]  → cacheScope: null     （不快取）
[static content]        → cacheScope: 'global' （全局快取！跨組織共享）
[dynamic content]       → cacheScope: null     （不快取）
```

這是快取效率最高的模式。`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 之前的靜態內容（Intro、Rules、Tone & Style 等）對所有用戶相同，可跨組織快取。

#### 模式 3：預設（3P 提供商 或 Boundary 缺失）

```
[attribution header]    → cacheScope: null     （不快取）
[system prompt prefix]  → cacheScope: 'org'    （組織級快取）
[everything else]       → cacheScope: 'org'    （組織級快取）
```

### `getCacheControl()`：TTL 決策

`src/services/api/claude.ts:359` 生成的 `cache_control` 對象：

```typescript
{
  type: 'ephemeral',
  ttl?: '1h',         // 僅特定 querySource 符合條件時
  scope?: 'global',   // 僅靜態區
}
```

1 小時 TTL 的判定邏輯（`should1hCacheTTL()`，第 394 行）：
- **Bedrock 用戶**：通過環境變量 `ENABLE_PROMPT_CACHING_1H_BEDROCK` 啓用
- **1P 用戶**：通過 GrowthBook 設定的 `allowlist` 數組匹配 `querySource`，支援前綴通配符（如 `"repl_main_thread*"`）
- **會話級鎖定**：資格判定結果在 bootstrap state 中快取，防止 GrowthBook 設定中途變化導致同一會話內 TTL 不一致

### 快取破壞：Session-Specific Guidance 的放置

`getSessionSpecificGuidanceSection()`（`src/constants/prompts.ts:352`）的內容必須放在 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` **之後**。因爲它包含：
- 當前會話的 enabledTools 集合
- `isForkSubagentEnabled()` 的執行時判定
- `getIsNonInteractiveSession()` 的結果

這些執行時 bit 如果放在靜態區，會產生 2^N 種 Blake2b 哈希變體（N = 執行時條件數），完全破壞快取命中率。源碼註釋明確警告：

> Each conditional here is a runtime bit that would otherwise multiply the Blake2b prefix hash variants (2^N). See PR #24490, #24171 for the same bug class.

## 上下文注入：System Context 與 User Context

System Prompt 數組本身不包含執行時上下文（git 狀態、CLAUDE.md 內容）。上下文通過兩個獨立的管道注入：

### System Context（`src/context.ts:116`）

```typescript
export const getSystemContext = memoize(async () => {
  return {
    gitStatus,           // git 分支、狀態、最近提交（截斷至 MAX_STATUS_CHARS=2000）
    cacheBreaker,        // 僅 ant 用戶的快取破壞器
  }
})
```

- 使用 `lodash.memoize` 快取——**整個會話期間只計算一次**
- Git 狀態快照包含 5 個並行 `git` 命令（branch、defaultBranch、status、log、userName）
- `status` 超過 2000 字符時截斷並附加提示使用 BashTool 取得更多信息
- `systemPromptInjection` 變更時，通過 `getUserContext.cache.clear?.()` 清除所有上下文快取

### User Context（`src/context.ts:155`）

```typescript
export const getUserContext = memoize(async () => {
  return {
    claudeMd,            // 合併後的 CLAUDE.md 內容
    currentDate,         // "Today's date is YYYY-MM-DD."
  }
})
```

- **CLAUDE.md 禁用條件**：`CLAUDE_CODE_DISABLE_CLAUDE_MDS` 環境變量，或 `--bare` 模式（除非通過 `--add-dir` 顯式指定目錄）
- `--bare` 模式的語義是"跳過我沒要求的東西"而非"忽略所有"

### 注入位置

在 `src/query.ts:449`：

```typescript
// System Context 追加到 System Prompt 尾部
const fullSystemPrompt = asSystemPrompt(
  appendSystemContext(systemPrompt, systemContext)  // 簡單拼接
)
```

User Context 通過 `prependUserContext()`（`src/utils/api.ts:449`）注入爲 `<system-reminder>` 標籤包裹的首條用戶訊息，放在所有對話訊息之前。

## Attribution Header：計費與安全

每個 API 請求的 System Prompt 首塊是 Attribution Header（`src/constants/system.ts:30`），包含：
- **`cc_version`**：Claude Code 版本 + 指紋
- **`cc_entrypoint`**：入口點標識（REPL / SDK / pipe 等）
- **`cch=00000`**（NATIVE_CLIENT_ATTESTATION 啓用時）：Bun 原生 HTTP 層在發送前將零替換爲計算出的哈希值，服務器驗證此 token 確認請求來自真實 Claude Code 客戶端

Header 始終 `cacheScope: null`——它因版本和指紋不同而變化，不適合快取。

## CLAUDE.md：專案級知識注入

這是 Claude Code 最巧妙的設計之一。在專案根目錄放一個 `CLAUDE.md` 文件，就能讓 AI "理解" 你的專案：

- **專案概述**：這個專案做什麼、用了什麼技術棧
- **開發約定**：程式碼風格、命名規範、分支策略
- **常用命令**：怎麼構建、怎麼測試、怎麼部署
- **注意事項**：已知的坑、特殊的設定

系統會自動發現併合並多級 CLAUDE.md：

```
~/.claude/CLAUDE.md              ← 用戶全局（個人偏好）
  └── /project/CLAUDE.md         ← 專案根目錄（團隊共享）
        └── /project/src/CLAUDE.md  ← 子目錄（模組特定）
```

加載邏輯在 `src/utils/claudemd.ts` 中的 `getClaudeMds()` 和 `getMemoryFiles()` 實現——從 CWD 向上遍歷目錄樹，合併所有匹配的 CLAUDE.md 檔案內容。

## 設計洞察：爲什麼是 `string[]` 而非單個 `string`

將 System Prompt 設計爲數組而非單段文本，是爲了 **快取分塊**：

1. Anthropic Prompt Cache 以 **內容塊**（TextBlock）爲快取單位
2. 將 System Prompt 拆爲多個塊，可以讓不變的部分（Intro、Rules）獲得獨立的快取命中
3. 如果是單個 `string`，任何一個字符變化（如日期更新）都會導致整個 System Prompt 的快取失效
4. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 標記允許 `splitSysPromptPrefix()` 精確地將靜態區標記爲 `scope: 'global'`，動態區不標記或標記爲 `scope: 'org'`

這是 Claude Code 在 token 成本優化上的核心設計——一次典型的 System Prompt 約 20K+ tokens，通過快取分塊可以節省 30-50% 的輸入 token 費用。
