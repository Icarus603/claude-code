## 爲什麼需要執行時 A/B 測試

構建時 `feature()` 是"全有或全無"的——要麼所有用戶都有，要麼所有用戶都沒有。但產品團隊需要更精細的控制：

- 只對 5% 的用戶灰度發佈新功能
- 按訂閱類型（Free / Pro / Team）差異化體驗
- 對特定組織靜默開啓實驗性能力
- 隨時遠程關閉出問題的功能，無需發版

這就是 **GrowthBook** 的用武之地——一個執行時的、基於用戶屬性的功能門控和 A/B 測試系統。

## 集成架構

GrowthBook 的完整實現位於 `src/services/analytics/growthbook.ts`（1156 行），工作流程如下：

<Steps>
  <Step title="啓動時取得遠程設定">
    CLI 啓動時，GrowthBook SDK 通過 `https://api.anthropic.com/` 的 API 端點取得當前的功能設定和實驗分組規則。使用 `remoteEval: true` 模式——在服務端計算分組，客戶端只拿結果。
  </Step>
  <Step title="計算用戶屬性">
    SDK 收集當前用戶的屬性（設備 ID、訂閱類型、組織 UUID 等），用於決定該用戶屬於哪些實驗的哪個分組。
  </Step>
  <Step title="快取到本地">
    計算結果快取到 `~/.claude.json` 的 `cachedGrowthBookFeatures` 字段。刷新間隔：Anthropic 員工 20 分鐘，外部用戶 6 小時。
  </Step>
  <Step title="程式碼中查詢 flag">
    業務程式碼通過 `tengu_*` 前綴的 flag 名查詢功能狀態，GrowthBook SDK 返回當前用戶的分組值。
  </Step>
</Steps>

## 用戶定向屬性

GrowthBook 根據以下用戶屬性決定實驗分組：

| 屬性 | 類型 | 來源 | 用途 |
|------|------|------|------|
| `id` | string | 會話 ID | 按會話粒度分組 |
| `deviceID` | string | 持久化設備標識 | 跨會話一致性 |
| `sessionId` | string | 當前會話 ID | 會話級實驗 |
| `platform` | enum | `process.platform` | 按操作系統差異化 |
| `organizationUUID` | string | API 認證信息 | 按組織灰度 |
| `accountUUID` | string | API 認證信息 | 按個人賬戶灰度 |
| `subscriptionType` | string | API 認證信息 | Free / Pro / Team 差異化 |
| `rateLimitTier` | string | API 認證信息 | 按速率限制層級 |
| `email` | string | API 認證信息 | 精確定向特定用戶 |
| `appVersion` | string | `MACRO.VERSION` | 按版本號灰度 |
| `github` | object | GitHub Actions 元資料 | CI 環境特殊處理 |

<Note>
這套定向系統意味着 Anthropic 可以做非常精細的實驗——比如"只對 Mac 上的 Pro 訂閱用戶的 10% 開啓新功能"。
</Note>

## 代號文化：tengu_* 的世界

所有執行時 flag 都以 `tengu_` 爲前綴——"Tengu"（天狗）是 Claude Code 的內部專案代號。flag 名採用**動物/植物/礦物 + 形容詞**的命名約定，刻意保持不透明。

<AccordionGroup>
  <Accordion title="tengu_kairos — Kairos 助手模式">
    控制 KAIROS 功能的執行時開關。即使構建時 `feature('KAIROS')` 通過，仍需此 flag 命中才能激活。雙重門控確保新功能可以分階段發佈。
  </Accordion>
  <Accordion title="tengu_amber_stoat — Explore Agent A/B 測試">
    控制內置的 Explore 子 Agent 的行爲變體。"amber stoat"（琥珀色白鼬）是隨機生成的代號，與功能內容無關——這是爲了防止通過 flag 名猜測功能。
  </Accordion>
  <Accordion title="tengu_auto_background_agents — 後臺 Agent 自動化">
    控制是否自動將某些任務分派給後臺 Agent 執行，而不是在前臺阻塞用戶。
  </Accordion>
  <Accordion title="tengu_onyx_plover — Auto-Dream 後臺記憶">
    控制"自動做夢"功能——在空閒時後臺整理和鞏固 Agent 的記憶。"onyx plover"（瑪瑙鴴）又是一個不透明代號。
  </Accordion>
  <Accordion title="tengu_glacier_2xr — 工具搜索行爲">
    控制 Tool Search 的行爲變體，可能是搜索算法或排序策略的 A/B 測試。
  </Accordion>
  <Accordion title="tengu_birch_trellis — Bash 權限策略">
    控制 BashTool 權限判斷的策略變體——可能在測試更寬鬆或更嚴格的權限規則。
  </Accordion>
  <Accordion title="tengu_scratch — 草稿本功能">
    控制一個實驗性的"草稿本"功能，可能是讓 AI 在處理複雜任務時使用中間暫存區。
  </Accordion>
  <Accordion title="tengu_quartz_lantern — Diff 計算策略">
    控制檔案寫入和編輯時的 diff 計算方式。可能在 A/B 測試不同的 diff 算法對用戶體驗的影響。
  </Accordion>
</AccordionGroup>

## Ant-Only 覆蓋機制

Anthropic 員工擁有兩種方式繞過 GrowthBook 的遠程求值：

### 環境變量覆蓋

```bash
# 僅在 USER_TYPE=ant 的構建中生效
CLAUDE_INTERNAL_FC_OVERRIDES='{"tengu_kairos": true}' claude
```

通過 `CLAUDE_INTERNAL_FC_OVERRIDES` 環境變量傳入 JSON 對象，直接覆蓋任意 flag 的值。

### Config 界面覆蓋

在內部構建中，`/config` 命令的 Gates 標籤頁提供了圖形化的 flag 管理界面，可以實時切換任意 GrowthBook flag。

## 實驗追蹤

GrowthBook 集成了完整的實驗曝光追蹤：

- 每次查詢 flag 時記錄實驗曝光事件
- 通過 protobuf 格式的 `GrowthbookExperimentEvent` 上報
- 包含 `variation_id`（0=對照組，1+=實驗組）和 `in_experiment` 標記
- 資料用於分析功能對用戶行爲的因果影響

<Note>
GrowthBook 正在從 Statsig 遷移而來——程式碼中仍保留着 `checkStatsigFeatureGate_CACHED_MAY_BE_STALE()` 這樣的遷移相容層。
</Note>
