## 一句話定義

Claude Code 是一個**執行在本地終端中的 agentic coding system**。它不是給建議的聊天機器人——它直接在你的專案目錄中讀程式碼、改文件、跑命令、調試程式，擁有完整的 shell 能力。

## 技術定位：terminal-native agentic system

理解 Claude Code 的關鍵在於三個詞：

| 定位關鍵詞 | 含義 |
|-----------|------|
| **Terminal-native** | 原生 CLI 應用，不是 IDE 擴充功能、不是 Web 界面、不是 API wrapper |
| **Agentic** | AI 自主決策工具呼叫鏈，不是"一問一答"的聊天模式 |
| **Coding system** | 面向軟件工程全流程，不是通用問答工具 |

與同類工具的**架構層面**差異（不是功能清單）：

| 工具 | 架構模式 | 執行位置 | 工具執行 |
|------|----------|----------|----------|
| **Claude Code** | Terminal-native agentic loop | 本地進程 | 直接 shell 執行 |
| Cursor / Copilot | IDE-integrated autocomplete + chat | IDE 進程內 | LSP / IDE API |
| Aider | CLI chat → git patch | 本地進程 | 檔案操作爲主 |
| ChatGPT / Claude.ai | Cloud chat + artifacts | 瀏覽器/雲端 | 沙箱容器 |

核心差異：Claude Code 擁有**完整的 shell 訪問權**——這意味着它可以做任何你在終端裏能做的事，但也需要對應的安全機制來約束這個能力。

## 端到端示例：從輸入到輸出

當你在終端中輸入 `bun run dev 有個 TypeScript 報錯，幫我修一下` 時，系統發生了什麼？

```
┌─────────────────────────────────────────────────────────┐
│ 1. 入口層 (cli.tsx → main.tsx)                          │
│    feature() = false, MACRO 注入, 啓動 Commander.js CLI  │
├─────────────────────────────────────────────────────────┤
│ 2. 交互層 (REPL.tsx — React/Ink)                        │
│    PromptInput 捕獲用戶輸入 → UserMessage 加入會話       │
├─────────────────────────────────────────────────────────┤
│ 3. 編排層 (QueryEngine.ts)                               │
│    管理 turn 生命週期、token 預算、compaction 觸發       │
├─────────────────────────────────────────────────────────┤
│ 4. 核心循環 (query.ts — Agentic Loop)                    │
│    組裝上下文 → 調 API → 收流式響應 → 解析工具呼叫      │
│    → 權限檢查 → 執行工具 → 結果回傳 → 再次調 API → 循環 │
├─────────────────────────────────────────────────────────┤
│ 5. 工具執行 (BashTool.call / FileEditTool.call / ...)    │
│    實際執行: 讀文件、執行命令、搜索程式碼...               │
├─────────────────────────────────────────────────────────┤
│ 6. 通信層 (claude.ts → Anthropic API)                    │
│    流式 HTTP, 支援 Bedrock/Vertex/Azure 多 provider      │
└─────────────────────────────────────────────────────────┘
```

具體到這個報錯修復場景，一次典型的 agentic loop 可能包含多輪工具呼叫：

| Turn | AI 決策 | 工具呼叫 | 結果 |
|------|---------|----------|------|
| 1 | 先看報錯信息 | `Bash("bun run dev 2>&1 | head -30")` | TypeScript 錯誤輸出 |
| 2 | 定位到文件 | `Read("src/utils/foo.ts")` | 源程式碼內容 |
| 3 | 搜索相關類型定義 | `Grep("interface Foo", "src/")` | 類型定義位置 |
| 4 | 修復程式碼 | `FileEdit(old, new)` | 程式碼已修改 |
| 5 | 驗證修復 | `Bash("bun run dev 2>&1 | head -10")` | 編譯通過 |

每一步都是 AI 自主決策的——它決定用哪個工具、傳什麼參數、何時停止。這就是 "agentic" 的含義。

## 它不是什麼

- **不是 IDE 擴充功能**：沒有圖形界面，不依賴 VS Code 或任何 IDE
- **不是 API wrapper**：它有自己的工具系統、權限模型、上下文工程、會話管理
- **不是聊天機器人**：輸出不是純文本，而是實際的文件修改、命令執行
- **不是無腦執行器**：每個敏感操作都有權限檢查和用戶確認環節

## 啓動入口解剖

真正的程式碼入口是 `src/entrypoints/cli.tsx`，它做了三件關鍵的事：

```typescript
// 1. 注入執行時 polyfill —— feature() 永遠返回 false
const feature = (_name: string) => false;

// 2. 注入構建時宏
globalThis.MACRO = { VERSION: "2.1.888", BUILD_TIME: ..., };

// 3. 聲明構建目標
globalThis.BUILD_TARGET = "external";  // 外部構建（非 Anthropic 內部）
globalThis.BUILD_ENV = "production";
globalThis.INTERFACE_TYPE = "stdio";   // 標準 I/O 交互
```

然後控制流傳遞到 `src/main.tsx`：
1. Commander.js 解析命令行參數
2. 初始化認證、遙測、策略限制
3. 加載工具列表（`getTools()`）
4. 啓動 REPL（`launchRepl()`）或管道模式（`-p`）

## 爲什麼選擇終端

終端不是限制，而是選擇。它帶來了獨特的能力：

- **完整的 shell 訪問**：可以執行任何命令行工具，無需爲每個能力撰寫擴充功能
- **專案原生**：直接在專案目錄工作，理解檔案系統結構、git 狀態
- **可組合性**：管道模式（`echo "..." | claude -p`）允許嵌入 CI/CD 和自動化流程
- **低延遲**：沒有 Electron 開銷，React/Ink 渲染的 TUI 響應極快

代價是用戶需要適應命令行界面——但也正因如此，它吸引的是需要**真正掌控開發環境**的開發者。
