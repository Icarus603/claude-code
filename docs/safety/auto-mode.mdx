## 概述

Auto mode 是 Claude Code 的一種權限模式，讓 AI 進入**連續自主執行**狀態。與傳統模式（每個敏感操作都彈出權限對話框等待用戶審批）不同，auto mode 使用 AI 分類器（transcript classifier）自動判斷每個工具呼叫是否安全，從而實現無中斷的執行體驗。

```
權限模式層級：

default  →  auto  →  bypassPermissions
（逐項確認） （AI 分類器審批） （全部放行）
```

## 核心架構

### 1. AI 分類器（yoloClassifier）

分類器是 auto mode 的核心，位於 `src/utils/permissions/yoloClassifier.ts`。

每個工具呼叫經過分類器評估，返回三種裁決之一：

| 裁決 | 含義 | 處理方式 |
|------|------|---------|
| **allow** | 操作安全 | 直接執行，用戶無感知 |
| **deny** | 操作危險 | 拒絕執行，向 AI 返回拒絕訊息 |
| **ask** | 無法確定 | 回退到傳統權限對話框 |

分類器的判斷基於完整的對話上下文（transcript），而不僅僅看單條命令——它能理解操作的意圖和上下文。

### 2. 兩階段分類流水線

分類器支援兩階段 XML 分類（`classifyYoloActionXml`）：

```
Stage 1 — "fast"（快速判斷）
├── max_tokens=64，stop_sequences 觸發即時 yes/no
├── 如果 allow → 直接放行（最快路徑）
└── 如果 block → 進入 Stage 2

Stage 2 — "thinking"（深度思考）
├── chain-of-thought 推理
├── 減少誤報（false positives）
└── 最終決定 allow / deny / ask
```

兩個階段共享相同的 system prompt 和 user content，利用 API 的 prompt caching（1 小時 TTL）優化性能。

可通過設定選擇模式：
- `'both'`（預設）— 兩階段都跑
- `'fast'` — 只跑 Stage 1
- `'thinking'` — 只跑 Stage 2

### 3. 分類器結果類型

```typescript
// src/types/permissions.ts
type YoloClassifierResult = {
  thinking?: string          // 分類器的推理過程
  shouldBlock: boolean       // 是否阻止
  reason: string             // 決策原因
  unavailable?: boolean      // 分類器是否不可用
  transcriptTooLong?: boolean // 對話是否超出上下文窗口
  model: string              // 使用的分類器模型
  stage?: 'fast' | 'thinking' // 哪個階段做出的決定
  // ... token 使用量、耗時等監控字段
}
```

## 安全機制

### 危險權限剝離

進入 auto mode 時，系統呼叫 `stripDangerousPermissionsForAutoMode()`（`permissionSetup.ts:510`），移除所有可能繞過分類器的 allow 規則。

被剝離的規則類型（`dangerousPatterns.ts`）：

| 規則類型 | 示例 | 剝離原因 |
|---------|------|---------|
| **Bash 程式碼執行** | `Bash(python:*)`, `Bash(node:*)` | 解釋器可執行任意程式碼，繞過分類器審查 |
| **Shell 入口** | `Bash(bash:*)`, `Bash(sh:*)` | 直接 shell 訪問等同無限制 |
| **Agent 規則** | `Agent(*)` | 任何 Agent allow 規則會繞過分類器審批子代理 |
| **PowerShell 程式碼執行** | `PowerShell(node:*)` | 同 Bash 邏輯 |
| **權限提升** | `Bash(sudo:*)`, `Bash(eval:*)` | 可執行任意命令 |

剝離的規則被暫存在 `strippedDangerousRules` 中，退出 auto mode 時通過 `restoreDangerousPermissions()` 恢復。

### 模型支援檢測

不是所有模型都支援 auto mode。`modelSupportsAutoMode()`（`src/utils/betas.ts`）檢查當前模型是否具備安全分類能力。不支援的模型無法進入 auto mode。

### Circuit Breaker 機制

`autoModeState.ts` 維護一個 circuit breaker 標誌：

```typescript
let autoModeCircuitBroken = false  // 由遠程設定控制
```

當遠程設定（GrowthBook `tengu_auto_mode_config.enabled`）設爲 `'disabled'` 時，circuit breaker 觸發，阻止 auto mode 的進入和繼續使用。這爲 Anthropic 提供了遠程緊急關停能力。

## 模式切換狀態管理

### 進入 Auto Mode

`transitionPermissionMode()`（`permissionSetup.ts:597`）處理所有模式切換：

```
1. 檢查 auto mode gate 是否開啓（isAutoModeGateEnabled）
2. 設置 autoModeActive = true
3. 呼叫 stripDangerousPermissionsForAutoMode() 剝離危險規則
4. 向對話注入 Auto Mode 系統提示
```

### 退出 Auto Mode

```
1. 設置 autoModeActive = false
2. 設置 needsAutoModeExitAttachment = true（觸發退出通知）
3. 呼叫 restoreDangerousPermissions() 恢復被剝離的規則
4. 向對話注入 "Exited Auto Mode" 提示
```

### 觸發路徑

Auto mode 可通過以下方式激活：
- CLI 參數 `--enable-auto-mode`
- settings.json 中的 `autoMode` 設定
- Plan mode 預設使用 auto mode 語義（`useAutoModeDuringPlan`，預設 true）
- SDK 控制訊息
- REPL 中 Shift+Tab 切換

## 系統提示詞

### 進入時（Full Instructions）

注入到對話中的指令（`messages.ts:3464`）：

> Auto mode is active. The user chose continuous, autonomous execution. You should:
>
> 1. **Execute immediately** — 直接實現，做合理假設
> 2. **Minimize interruptions** — 常規決策自行判斷，減少提問
> 3. **Prefer action over planning** — 預設直接編碼，不進 plan mode
> 4. **Expect course corrections** — 用戶可隨時糾正
> 5. **Do not take overly destructive actions** — 刪除資料/修改生產系統仍需確認
> 6. **Avoid data exfiltration** — 不主動分享密鑰/內部文件

### 持續執行時（Sparse Instructions）

後續輪次注入簡短提醒：

> Auto mode still active. Execute autonomously, minimize interruptions, prefer action over planning.

### 退出時（Exit Instructions）

> You have exited auto mode. Ask clarifying questions when the approach is ambiguous rather than making assumptions.

## 與 Plan Mode 的協作

Plan mode 預設使用 auto mode 語義（`getUseAutoModeDuringPlan()`，預設 true）。這意味着：

- Plan mode 進入時，如果 auto mode 可用，也會激活分類器
- `isAutoModeActive()` 是權威信號（`prePlanMode`/`strippedDangerousRules` 不可靠）
- 退出 plan mode 時會同時退出 auto mode

## 分類器不可用的降級策略

當分類器 API 不可用時（`unavailable: true` 或 `transcriptTooLong: true`）：

- 不會直接 allow — 回退到傳統的權限對話框（ask）
- 向 AI 發送訊息："{model} is temporarily unavailable, so auto mode cannot determine the safety of {toolName} right now."
- 確定性錯誤（如對話過長）不重試，直接降級

## 分類器 Prompt 模板

分類器的行爲由三個 prompt 文件控制，位於 `src/utils/permissions/yolo-classifier-prompts/`。這些文件在構建時通過 `require()` 內聯爲字符串常量，執行時不可修改。

### auto_mode_system_prompt.txt

主系統提示詞，定義分類器的角色、分類流程和決策類別。包含：

- **分類流程**：理解操作 → 檢查用戶意圖 → 評估風險
- **BLOCK 始終阻止**：外部程式碼執行、不可逆刪除、未授權持久化、安全削弱、提權、網絡服務
- **BLOCK 除非明確意圖**：CWD 外寫入、系統包管理、git push、大規模變更
- **ALLOW 安全操作**：讀文件、搜索、git 只讀命令、測試/lint/構建、CWD 內編輯
- `<permissions_template>` 佔位符 — 執行時替換爲具體權限模板（external 或 anthropic）
- 末尾 `Use the classify_result tool to report your classification.` — XML 模式下被替換爲 `<block>` 標籤格式說明

### permissions_external.txt

外部用戶版本的權限模板。三個 `<user_*_to_replace>` 標籤內包裹預設規則（bullet 格式），用戶自定義規則**整體替換**預設值：

```
<user_allow_rules_to_replace>
- 預設 allow 規則 1
- 預設 allow 規則 2
</user_allow_rules_to_replace>
```

- **allow**：9 條預設規則（只讀命令、版本檢查、git 只讀、測試/lint/構建、包安裝、CWD 內檔案操作等）
- **soft_deny**：10 條預設規則（外部程式碼執行、遞歸刪除、shell 設定修改、提權、網絡服務等）
- **environment**：4 條環境描述（終端環境、auto mode 上下文、開發工具可用、語言/框架不限）

`getDefaultExternalAutoModeRules()` 從此文件提取 bullet 項，供 `claude auto-mode defaults` 命令輸出。

### permissions_anthropic.txt

Anthropic 內部版本的權限模板。預設規則在標籤**外部**，標籤內爲空，用戶自定義規則以**追加**方式疊加：

```
- 預設規則（在標籤外，始終生效）
<user_allow_rules_to_replace>
</user_allow_rules_to_replace>
```

相比 external 版本，額外包含：
- 雲 CLI 只讀命令（aws describe, gcloud describe, kubectl get 等）
- 基礎設施即程式碼 plan 命令（terraform plan, pulumi preview 等）
- 對應的 deny 規則（雲資源建立/修改/刪除、IaC apply、生產環境訪問等）

### 模板替換流程

```
buildYoloSystemPrompt()
  ├── BASE_PROMPT.replace('<permissions_template>', EXTERNAL/ANTHROPIC_TEMPLATE)
  ├── .replace(<user_allow_rules_to_replace>, userAllow ?? defaults)
  ├── .replace(<user_deny_rules_to_replace>, userDeny ?? defaults)
  └── .replace(<user_environment_to_replace>, userEnvironment ?? defaults)
```

- 外部模板：用戶設置非空時**替換**對應標籤內容，否則保留預設值
- 內部模板：用戶設置**追加**到預設值之後（標籤在末尾爲空）

## 當前狀態說明

> **注意**：auto mode 的完整程式碼邏輯已存在於程式碼庫中，但依賴 `feature('TRANSCRIPT_CLASSIFIER')` feature flag。
> 在當前反編譯版本中，`feature()` 始終返回 `false`，因此 auto mode 不可用。
> 要啓用需將 `feature('TRANSCRIPT_CLASSIFIER')` 改爲 `true`，並確保 GrowthBook 設定源有合理的 fallback 預設值。

Prompt 模板文件爲**重建產物**——原始文件在反編譯過程中丟失，已根據程式碼邏輯和 `yoloClassifier.ts` 中的替換模式重新編寫。
