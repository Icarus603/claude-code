## 問題場景

你說"重構這個模組"，AI 立刻開始改程式碼——但你還沒搞清楚它打算怎麼改。等改了一半發現方向不對，已經來不及了。

## Plan Mode 的解決方案

計劃模式給對話加了一個"只讀階段"，通過兩個工具實現閉環：

<Steps>
  <Step title="EnterPlanMode — 進入計劃模式">
    AI 自主判斷（或用戶觸發）任務需要規劃，呼叫 `EnterPlanModeTool`（`src/tools/EnterPlanModeTool/EnterPlanModeTool.ts:36`）。該工具需要**用戶審批**（`checkPermissions` 返回 `ask`）。
  </Step>
  <Step title="探索階段 — 只讀工具集">
    權限模式切換爲 `'plan'`，AI 只能使用 `isReadOnly()` 爲 true 的工具（Read、Grep、Glob、Agent 等）。寫操作被自動拒絕。
  </Step>
  <Step title="ExitPlanMode — 提交方案審批">
    AI 完成探索後，呼叫 `ExitPlanModeV2Tool`（`src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts:147`），將計劃文件提交給用戶審閱。這是第二個**需要用戶審批**的節點。
  </Step>
  <Step title="恢復執行 — 全部工具權限">
    用戶批准後，權限模式恢復爲進入前的狀態，AI 按計劃執行。
  </Step>
</Steps>

## 權限的自動收窄與恢復

### 進入：`prepareContextForPlanMode()`

`EnterPlanModeTool.call()`（第 77 行）的核心邏輯：

```typescript
// 1. 記錄轉換狀態（保存之前的模式）
handlePlanModeTransition(currentMode, 'plan')

// 2. 切換權限上下文爲 plan 模式
context.setAppState(prev => ({
  ...prev,
  toolPermissionContext: applyPermissionUpdate(
    prepareContextForPlanMode(prev.toolPermissionContext),
    { type: 'setMode', mode: 'plan', destination: 'session' },
  ),
}))
```

`prepareContextForPlanMode()`（`src/utils/permissions/permissionSetup.ts`）做了什麼：
- 建立新的 `ToolPermissionContext`，`mode` 設爲 `'plan'`
- 在 plan 模式下，工具的 `isReadOnly()` 檢查成爲唯一準入條件
- 如果用戶的預設模式是 `'auto'`，還會激活 classifier 的副作用

### 退出：權限恢復 + Prompt-based 權限

`ExitPlanModeV2Tool` 的退出邏輯做了兩件關鍵的事：

**1. 恢復權限模式**

通過 `handlePlanModeTransition()` 和 `applyPermissionUpdate()` 恢復到進入前的模式。

**2. 注入 Prompt-based 權限**

這是 Plan Mode 最精妙的設計——AI 可以在計劃中聲明它需要執行的命令類別：

```typescript
// ExitPlanModeV2Tool 的 inputSchema
allowedPrompts: z.array(z.object({
  tool: z.enum(['Bash']),
  prompt: z.string().describe('Semantic description, e.g. "run tests"'),
})).optional()
```

當 AI 提交計劃時，如果聲明瞭 `allowedPrompts: [{ tool: 'Bash', prompt: 'run tests' }]`，用戶批准後，"run tests" 這類 Bash 命令會被自動放行——不再需要逐個確認。

## 計劃文件的持久化

計劃內容被寫入磁盤文件（由 `getPlanFilePath()` 確定路徑），這與簡單的"AI 說一段話然後開始執行"有本質區別：

1. `ExitPlanModeV2Tool` 的 `normalizeToolInput` 從磁盤讀取計劃內容，注入到 `input.plan` 和 `input.planFilePath`
2. 計劃文件是用戶**可編輯**的——用戶可以在審批前修改 AI 的方案
3. `planWasEdited` 字段標記用戶是否修改了計劃，影響後續的 tool_result 回顯
4. `persistFileSnapshotIfRemote()` 在遠程場景下保存檔案快照

## Teammate 場景下的計劃審批

在 Agent Swarms（`isAgentSwarmsEnabled()`）模式下，計劃審批有額外的協作流程：

```typescript
// 如果是 Teammate 角色
if (isTeammate()) {
  // 發送計劃到 Team Leader 的 mailbox 等待審批
  const requestId = generateRequestId()
  writeToMailbox(getTeamName(), {
    type: 'plan_approval_request',
    plan, requestId, ...
  })
  // 返回 awaitingLeaderApproval: true
  // Team Leader 審批後通過 mailbox 通知 Teammate
}
```

這意味着在蜂羣模式下，計劃可能不是由直接用戶審批，而是由 Team Leader 審批。

## 什麼時候該用計劃模式

`EnterPlanModeTool` 的 Prompt（`src/tools/EnterPlanModeTool/prompt.ts`）定義了兩套觸發標準——外部版本更積極（鼓勵規劃），內部版本更剋制（僅在真正模糊時使用）：

| 場景 | 外部版本 | 內部版本 |
|------|---------|---------|
| 修復 typo | 跳過 | 跳過 |
| 添加刪除按鈕 | **進入**（涉及多個文件） | **跳過**（路徑明確） |
| 重構認證系統 | **進入** | **進入**（高影響重構） |
| "開始做 X" | — | **跳過**（直接開始） |
| 架構決策（Redis vs 內存快取） | **進入** | **進入**（真正模糊） |

## 計劃模式 + 任務系統

計劃模式通常與任務系統配合使用：

1. 在計劃模式中，AI 把實施步驟建立爲任務列表（`TodoWrite`）
2. 用戶審批計劃（包含任務列表）
3. 退出計劃模式後，AI 按任務列表逐項執行
4. 用戶可以通過任務列表追蹤進度

## 完整生命週期

```
用戶: "重構這個模組"
  ↓
AI 判斷需要規劃 → 呼叫 EnterPlanModeTool
  ↓ 用戶審批（Ask 對話框）
handlePlanModeTransition(default, 'plan')  // 保存 default
prepareContextForPlanMode()                // 建立只讀上下文
  ↓
AI 使用 Read/Grep/Glob/Agent 探索程式碼庫
  ↓ （可能 10+ 輪只讀工具呼叫）
AI 形成方案 → 呼叫 ExitPlanModeV2Tool({
  allowedPrompts: [
    { tool: 'Bash', prompt: 'run tests' },
    { tool: 'Bash', prompt: 'install dependencies' }
  ]
})
  ↓ 用戶審批計劃（可編輯計劃文件）
恢復權限模式 → 注入 prompt-based 權限
  ↓
AI 使用全部工具執行計劃，"run tests" 等命令自動放行
```
