## 6 種 Hook 類型

Hooks 設定支援 6 種執行方式（`src/types/hooks.ts`）：

| 類型 | 執行方式 | 適用場景 |
|------|---------|---------|
| `command` | Shell 命令（bash/PowerShell） | 通用腳本、CI 檢查 |
| `prompt` | 注入到 AI 上下文 | 程式碼規範提醒 |
| `agent` | 啓動子 Agent 執行 | 複雜分析任務 |
| `http` | HTTP 請求 | 遠程服務、Webhook |
| `callback` | 內部 JS 函數 | 系統內置 Hook |
| `function` | 執行時註冊的函數 Hook | Agent/Skill 內部使用 |

## 執行引擎：execCommandHook

`execCommandHook()`（`src/utils/hooks.ts:829-1417`）是命令型 Hook 的執行核心：

```
execCommandHook(hook, hookEvent, hookName, jsonInput, signal)
  ├── Shell 選擇: hook.shell ?? DEFAULT_HOOK_SHELL
  │   ├── bash: spawn(cmd, [], { shell: gitBashPath | true })
  │   └── powershell: spawn(pwsh, ['-NoProfile', '-NonInteractive', '-Command', cmd])
  ├── 變量替換
  │   ├── ${CLAUDE_PLUGIN_ROOT} → pluginRoot 路徑
  │   ├── ${CLAUDE_PLUGIN_DATA} → plugin 資料目錄
  │   └── ${user_config.X} → 用戶設定值
  ├── 環境變量注入
  │   ├── CLAUDE_PROJECT_DIR
  │   ├── CLAUDE_ENV_FILE（SessionStart/Setup/CwdChanged/FileChanged）
  │   └── CLAUDE_PLUGIN_OPTION_*（plugin options）
  ├── stdin 寫入: jsonInput + '\n'
  ├── 超時: hook.timeout * 1000 ?? 600000ms（10分鐘）
  └── 異步檢測: 檢查 stdout 首行是否爲 {"async":true}
```

### 異步 Hook 的檢測協議

Hook 進程的 stdout 第一行如果是 `{"async":true}`，系統將其轉爲後臺任務（`hooks.ts:1199-1246`）：

```typescript
const firstLine = firstLineOf(stdout).trim()
if (isAsyncHookJSONOutput(parsed)) {
  executeInBackground({
    processId: `async_hook_${child.pid}`,
    asyncResponse: parsed,
    ...
  })
}
```

後臺 Hook 通過 `registerPendingAsyncHook()` 註冊到 `AsyncHookRegistry`，完成後通過 `enqueuePendingNotification()` 通知主執行緒。

### asyncRewake：Hook 喚醒模型

`asyncRewake` 模式的 Hook 繞過 `AsyncHookRegistry`。當 Hook 退出碼爲 2 時，通過 `enqueuePendingNotification()` 以 `task-notification` 模式注入訊息，喚醒空閒的模型（通過 `useQueueProcessor`）或在忙碌時注入 `queued_command` 附件。

## Hook 輸出的 JSON Schema

同步 Hook 的輸出遵循嚴格的 Zod schema（`src/types/hooks.ts:49-567`）：

```json
{
  "continue": false,                    // 是否繼續執行
  "suppressOutput": true,               // 隱藏 stdout
  "stopReason": "安全檢查失敗",           // continue=false 時的原因
  "decision": "approve" | "block",      // 全局決策
  "reason": "原因說明",                   // 決策原因
  "systemMessage": "警告內容",           // 注入到上下文的系統訊息
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow" | "deny" | "ask",
    "permissionDecisionReason": "匹配了安全規則",
    "updatedInput": { ... },            // 修改後的工具輸入
    "additionalContext": "額外上下文"     // 注入到對話
  }
}
```

### 各事件的 hookSpecificOutput

| 事件 | 專有字段 | 作用 |
|------|---------|------|
| `PreToolUse` | `permissionDecision`, `updatedInput`, `additionalContext` | 攔截/修改工具輸入 |
| `UserPromptSubmit` | `additionalContext` | 注入額外上下文 |
| `PostToolUse` | `additionalContext`, `updatedMCPToolOutput` | 修改 MCP 工具輸出 |
| `SessionStart` | `initialUserMessage`, `watchPaths` | 設置初始訊息和文件監控 |
| `PermissionDenied` | `retry` | 指示是否重試 |
| `Elicitation` | `action`, `content` | 控制用戶輸入對話框 |

## Hook 匹配機制：getMatchingHooks

`getMatchingHooks()`（`hooks.ts:1685-1956`）負責從所有來源中查找匹配的 Hook：

### 多來源合併

```
getHooksConfig()
  ├── getHooksConfigFromSnapshot()    ← settings.json 中的 Hook（user/project/local）
  ├── getRegisteredHooks()            ← SDK 註冊的 callback Hook
  ├── getSessionHooks()               ← Agent/Skill 前置註冊的 session Hook
  └── getSessionFunctionHooks()       ← 執行時 function Hook
```

### 匹配規則

`matcher` 字段支援三種模式（`matchesPattern()`, `hooks.ts:1428-1463`）：

```
"Write"              → 精確匹配
"Write|Edit"         → 管道分隔的多值匹配
"^Bash(git.*)"       → 正則匹配
"*" 或 ""            → 通配（匹配所有）
```

### if 條件過濾

Hook 可以指定 `if` 條件，只在特定輸入時觸發。`prepareIfConditionMatcher()`（`hooks.ts:1472-1503`）預編譯匹配器：

```json
{
  "hooks": [{
    "command": "check-git-branch.sh",
    "if": "Bash(git push*)"
  }]
}
```

`if` 條件使用 `permissionRuleValueFromString` 解析，支援與權限規則相同的語法（工具名 + 參數模式）。Bash 工具還會使用 tree-sitter 進行 AST 級別的命令解析。

### Hook 去重

同一個 Hook 命令在不同設定層級（user/project/local）可能重複。系統按 `pluginRoot\0command` 做 Map 去重，保留**最後合併的層級**。

## 工作區信任檢查

**所有 Hook 都要求工作區信任**（`shouldSkipHookDueToTrust()`, `hooks.ts:286-296`）。這是縱深防禦措施——防止惡意倉庫的 `.claude/settings.json` 在未信任的情況下執行任意命令。

```typescript
// 交互模式下，所有 Hook 要求信任
const hasTrust = checkHasTrustDialogAccepted()
return !hasTrust
```

SDK 非交互模式下信任是隱式的（`getIsNonInteractiveSession()` 爲 true 時跳過檢查）。

## 四種 Hook 能力的源碼映射

### 1. 攔截操作（PreToolUse）

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny"
  }
}
```

`processHookJSONOutput()` 將 `permissionDecision` 映射爲 `result.permissionBehavior = 'deny'`，並設置 `blockingError`，阻止工具執行。

### 2. 修改行爲（updatedInput / updatedMCPToolOutput）

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "updatedInput": { "command": "npm test -- --bail" }
  }
}
```

`updatedInput` 替換原始工具輸入；`updatedMCPToolOutput`（PostToolUse 事件）替換 MCP 工具的返回值——可用於過濾敏感資料。

### 3. 注入上下文（additionalContext / systemMessage）

- `additionalContext` → 通過 `createAttachmentMessage({ type: 'hook_additional_context' })` 注入爲用戶訊息
- `systemMessage` → 注入爲系統警告，直接顯示給用戶

### 4. 控制流程（continue / stopReason）

```json
{ "continue": false, "stopReason": "構建失敗，停止執行" }
```

`continue: false` 設置 `preventContinuation = true`，阻止 Agent 繼續執行後續操作。

## Session Hook 的生命週期

Agent 和 Skill 的前置 Hook 通過 `registerFrontmatterHooks()` 註冊（`runAgent.ts:567-575`），綁定到 agent 的 session ID。Agent 結束時通過 `clearSessionHooks()` 清理。

```typescript
// runAgent.ts:567 — 註冊 agent 的前置 Hook
registerFrontmatterHooks(rootSetAppState, agentId, agentDefinition.hooks, ...)

// runAgent.ts:820 — finally 塊清理
clearSessionHooks(rootSetAppState, agentId)
```

這確保 Agent A 的 Hook 不會泄漏到 Agent B 的執行中。
