## 單輪 vs 多輪：架構層面的差異

- **單輪**（一次 Agentic Loop）：`query()` 函數的一次完整執行——組裝上下文 → 調 API → 處理工具呼叫 → 循環直到結束
- **多輪**（一個 Session）：`QueryEngine` 類管理的一次會話——跨越數十輪 `submitMessage()` 呼叫，持續數小時

`QueryEngine`（`src/QueryEngine.ts:186`）是單輪 Agentic Loop 之上的**會話編排器**，它管理的狀態遠不止訊息列表：

```
QueryEngine 內部狀態
├── mutableMessages: Message[]         ← 完整對話歷史，跨 turn 累積
├── readFileState: FileStateCache      ← 已讀檔案內容快取，避免重複讀取
├── totalUsage: NonNullableUsage       ← 累計 token 消耗（input/output/cache）
├── permissionDenials: SDKPermissionDenial[]  ← 權限拒絕記錄
├── discoveredSkillNames: Set<string>  ← 當前 turn 已發現的 skill
└── abortController: AbortController   ← 會話級中斷控制
```

## QueryEngine 的核心方法：submitMessage()

每次用戶輸入一條訊息，REPL 或 SDK 呼叫 `submitMessage()`，它會執行完整的 turn 初始化鏈路：

```typescript
// src/QueryEngine.ts:211 — 簡化的 submitMessage 流程
async *submitMessage(prompt, options?): AsyncGenerator<SDKMessage> {
  // 1. 清除 turn 級追蹤狀態
  this.discoveredSkillNames.clear()
  
  // 2. 解析模型（用戶可能中途切換了模型）
  const mainLoopModel = userSpecifiedModel
    ? parseUserSpecifiedModel(userSpecifiedModel)
    : getMainLoopModel()
  
  // 3. 動態組裝 System Prompt（每次 turn 都重新構建）
  const { defaultSystemPrompt, userContext, systemContext } =
    await fetchSystemPromptParts({ tools, mainLoopModel, mcpClients })
  
  // 4. 包裝權限檢查（追蹤每次拒絕）
  const wrappedCanUseTool = async (tool, input, ...) => {
    const result = await canUseTool(tool, input, ...)
    if (result.behavior !== 'allow') {
      this.permissionDenials.push({ tool_name: tool.name, ... })
    }
    return result
  }
  
  // 5. 呼叫核心 query() 函數執行 agentic loop
  yield* query({
    systemPrompt, messages: this.mutableMessages,
    tools, model: mainLoopModel, ...
  })
}
```

關鍵設計：`submitMessage()` 是 `async *Generator`——它逐步 yield `SDKMessage`，讓呼叫方（REPL/SDK）能實時展示進度，而不是等整個 turn 結束。

## 會話持久化：JSONL Transcript

每次對話事件都被追加寫入 transcript 文件（`src/utils/sessionStorage.ts`）：

### 存儲路徑

```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

- `project-hash` 由 `getProjectDir(originalCwd)` 生成，同一專案目錄的會話歸入同一子目錄
- 每條記錄是一行 JSON（JSONL 格式），支援追加寫入而不需要讀取-修改-寫入整個文件
- 讀取上限爲 50MB（`MAX_TRANSCRIPT_READ_BYTES`），防止超大會話導致 OOM

### Transcript 寫入器

`TranscriptWriter`（`src/utils/sessionStorage.ts:1200+`）是一個寫隊列，確保併發的訊息追加不會互相覆蓋：

```
寫入流程：
  appendEntryToFile(sessionId, entry)
    ↓
  ensureCurrentSessionFile()   ← 懶初始化：首次寫入時才建立文件
    ↓
  序列化爲 JSON + 換行符
    ↓
  appendFile(path, line)       ← 原子追加
    ↓
  如果設定了遠程持久化：
    persistToRemote(sessionId, entry)
      ├── CCR v2: internalEventWriter('transcript', entry)
      └── v1 Ingress: sessionIngress.appendSessionLog(...)
```

### 會話恢復鏈路

`--resume` 參數觸發的恢復流程（`src/main.tsx:3620+`）：

```
1. 解析 resume 參數：
   ├── UUID 格式 → getTranscriptPathForSession(uuid)
   ├── .jsonl 檔案路徑 → 直接使用
   └── boolean → 最近一次會話的 picker
   
2. loadTranscriptFromFile(path)
   ├── 按 JSONL 行解析
   ├── 過濾出訊息類型記錄
   └── 重建 Message[] 數組

3. 恢復上下文狀態：
   ├── restoreCostStateForSession(sessionId)  ← 恢復累計費用
   ├── 恢復 agentSetting（用戶選擇的 Agent 類型）
   └── 如果有 --rewind-files，恢復文件到指定訊息時的快照

4. 建立 QueryEngine({ initialMessages: restoredMessages })
   └── 從恢復的訊息繼續對話
```

## 成本追蹤：從 API Usage 到美元

成本追蹤貫穿三個模組，形成完整的記錄→累計→展示鏈路：

### 記錄層：API 響應中的 Usage

每個 `message_delta` 事件攜帶 `usage` 字段（`input_tokens`、`output_tokens`、`cache_creation_input_tokens`、`cache_read_input_tokens`）。`accumulateUsage()` 將增量 usage 累加到會話總量。

### 累計層：cost-tracker.ts

```typescript
// src/cost-tracker.ts — StoredCostState 資料模型
type StoredCostState = {
  totalCostUSD: number                       // 累計美元花費
  totalAPIDuration: number                   // API 呼叫總時長（含重試）
  totalAPIDurationWithoutRetries: number     // 不含重試的純推理時間
  totalToolDuration: number                  // 工具執行總時長
  totalLinesAdded: number                    // 程式碼增加行數
  totalLinesRemoved: number                  // 程式碼刪除行數
  modelUsage: { [modelName: string]: ModelUsage }  // 按模型分拆的用量
}
```

`addToTotalSessionCost()` 根據模型定價計算每次 API 呼叫的費用，累計到 `totalCostUSD`。按模型的 `ModelUsage` 支援在同一會話中切換模型後分別統計。

### 持久化：跨重啓保留

```typescript
// 每次會話結束時保存到專案設定
saveCurrentSessionCosts(sessionId)
  → projectConfig.lastCost = totalCostUSD
  → projectConfig.lastSessionId = sessionId
  → projectConfig.lastModelUsage = modelUsage
```

### 預算熔斷

`QueryEngineConfig.maxBudgetUsd` 提供了會話級的硬性預算上限。在 REPL 中，當累計費用超過 $5 時（`src/screens/REPL.tsx:2208`），彈出費用提醒對話框——這不是硬性阻斷，而是"軟提醒"。

## 模型熱切換

在一個會話中切換模型不會丟失對話歷史——因爲 `mutableMessages` 與模型選擇是解耦的：

```
/model sonnet → setMainLoopModelOverride('claude-sonnet-4-20250514')
  ↓
下一次 submitMessage() 開始時：
  ↓
parseUserSpecifiedModel(userSpecifiedModel)
  → 返回新的模型設定
  ↓
fetchSystemPromptParts({ mainLoopModel: newModel })
  → System Prompt 根據新模型能力重新組裝
  ↓
query({ model: newModel, messages: this.mutableMessages })
  → 使用完整歷史 + 新模型繼續對話
```

切換模型時，`contextWindowTokens` 和 `maxOutputTokens` 也會根據新模型的規格重新計算——例如從 Sonnet 切換到 Opus 時，上下文窗口可能從 200K 變爲 1M。

## 檔案快照與回滾

`fileHistoryMakeSnapshot()`（`src/utils/fileHistory.ts`）在 AI 每次修改文件前自動保存當前內容。快照綁定到具體的 `message.id`，使得 `--rewind-files <user-message-id>` 可以精確恢復到對話中任意時間點的文件狀態——這比 git 更細粒度（git 只追蹤已提交的內容）。
