## 權限之外的第二道防線

權限系統決定"這條命令能不能執行"，沙箱決定"執行時能做到什麼程度"。

即使一條命令通過了權限審批，沙箱仍然可以限制它的行爲。兩者構成縱深防禦的兩層：
- **權限層**（應用級）：在工具呼叫前檢查，決定是否彈窗審批
- **沙箱層**（OS 級）：在進程級別強制約束，即使 AI 生成了惡意命令也無法突破

## 執行鏈路：從用戶輸入到沙箱包裹

一條 Bash 命令的完整執行路徑如下：

```
用戶輸入 → BashTool.call()
         → shouldUseSandbox(input) ─── 是否需要沙箱？
         → Shell.exec(command, { shouldUseSandbox })
         → SandboxManager.wrapWithSandbox(command)
         → spawn(wrapped_command)  ─── 實際進程建立
```

關鍵判定發生在 `shouldUseSandbox()`（`src/tools/BashTool/shouldUseSandbox.ts`），它執行以下檢查：

1. **全局開關**：`SandboxManager.isSandboxingEnabled()` — 檢查平臺支援 + 依賴完整性 + 用戶設置
2. **顯式跳過**：如果 `dangerouslyDisableSandbox: true` 且策略允許（`allowUnsandboxedCommands`），則不走沙箱
3. **排除列表**：用戶可在 `settings.json` 中設定 `sandbox.excludedCommands`，匹配的命令跳過沙箱
4. **預設行爲**：以上條件都不滿足時，**進入沙箱**

## `shouldUseSandbox()` 判定邏輯詳解

```typescript
// src/tools/BashTool/shouldUseSandbox.ts
function shouldUseSandbox(input: Partial<SandboxInput>): boolean {
  // 1. 全局未啓用 → 直接跳過
  if (!SandboxManager.isSandboxingEnabled()) return false

  // 2. 顯式禁用 + 策略允許 → 跳過
  if (input.dangerouslyDisableSandbox && 
      SandboxManager.areUnsandboxedCommandsAllowed()) return false

  // 3. 無命令 → 跳過
  if (!input.command) return false

  // 4. 匹配排除列表 → 跳過
  if (containsExcludedCommand(input.command)) return false

  // 5. 其他情況 → 必須沙箱化
  return true
}
```

`containsExcludedCommand()` 的匹配機制值得注意——它不只是簡單的前綴匹配，而是支援三種模式：

| 模式 | 示例 | 匹配行爲 |
|------|------|----------|
| **精確匹配** | `npm run lint` | 完全相等 |
| **前綴匹配** | `npm run test:*` | 前綴 + 空格或完全相等 |
| **通配符** | `docker*` | 使用 `matchWildcardPattern` |

對於複合命令（如 `docker ps && curl evil.com`），系統會先拆分爲子命令，逐一檢查。還會迭代剝離環境變量前綴（`FOO=bar bazel ...`）和包裝命令（`timeout 30 bazel ...`），直到不動點——防止通過嵌套包裝繞過。

## 沙箱的設定模型

沙箱設定來自 `settings.json` 中的 `sandbox` 字段（`src/entrypoints/sandboxTypes.ts`）：

```jsonc
{
  "sandbox": {
    "enabled": true,                     // 主開關
    "autoAllowBashIfSandboxed": true,     // 沙箱中的命令自動允許（跳過審批）
    "allowUnsandboxedCommands": true,     // 是否允許 dangerouslyDisableSandbox
    "failIfUnavailable": false,           // 沙箱依賴缺失時是否報錯退出
    
    "network": {
      "allowedDomains": ["github.com"],   // 網絡白名單
      "deniedDomains": [],                // 網絡黑名單
      "allowLocalBinding": true,          // 允許 localhost 綁定
      "httpProxyPort": 8888               // HTTP 代理端口（MITM）
    },
    
    "filesystem": {
      "allowWrite": ["~/projects"],       // 額外可寫路徑
      "denyWrite": ["~/.ssh"],            // 禁止寫入路徑
      "denyRead": [],                     // 禁止讀取路徑
      "allowRead": []                     // 在 denyRead 中重新放行
    },
    
    "excludedCommands": ["docker", "npm:*"]  // 不走沙箱的命令
  }
}
```

`SandboxSettingsSchema` 定義了完整的 Zod 驗證規則，包含一些未公開的設置如 `enabledPlatforms`（限制沙箱只在特定平臺生效）。

## 平臺實現差異

### macOS：sandbox-exec（Seatbelt）

macOS 使用 Apple 的 Seatbelt 沙箱（`sandbox-exec` 命令），這是 macOS 原生的進程隔離機制。

執行流程：
1. `SandboxManager.wrapWithSandbox()` 呼叫 `@anthropic-ai/sandbox-runtime` 的 `BaseSandboxManager`
2. 執行時生成 Seatbelt profile（基於設定中的網絡/檔案系統規則）
3. 通過 `sandbox-exec -p <profile> -- <command>` 包裹原始命令
4. Seatbelt 在內核級別強制執行約束

網絡隔離的實現方式：
- 通過代理端口攔截 HTTP/HTTPS 請求
- 域名白名單/黑名單在代理層過濾
- Unix socket 可單獨設定允許路徑

### Linux：bubblewrap（bwrap）+ seccomp

Linux 使用 `bubblewrap`（bwrap）建立命名空間隔離，配合 seccomp 過濾系統呼叫：

依賴項（`apt install`）：
| 包 | 作用 |
|----|------|
| `bubblewrap` | 建立 mount/PID/network 命名空間 |
| `socat` | 網絡代理（HTTP/SOCKS） |
| `libseccomp` / seccomp filter | 過濾 Unix socket 系統呼叫 |

bwrap 的實現差異：
- **不支援 glob 路徑模式**（macOS 的 Seatbelt 支援）— Linux 上帶 glob 的權限規則會觸發警告
- 執行後會在當前目錄留下 0 字節的 mount-point 文件（如 `.bashrc`），需要 `cleanupAfterCommand()` 清理
- seccomp 無法按路徑過濾 Unix socket（只能全允許或全拒絕），與 macOS 的按路徑放行形成差異

### 平臺支援矩陣

| 特性 | macOS | Linux | WSL |
|------|-------|-------|-----|
| 沙箱引擎 | sandbox-exec (Seatbelt) | bubblewrap + seccomp | 僅 WSL2 |
| 文件 glob | ✅ 完整支援 | ⚠️ 僅 `/**` 後綴 | 同 Linux |
| 網絡 Unix socket 按路徑 | ✅ | ❌ | ❌ |
| 依賴檢查 | ripgrep | bwrap + socat + ripgrep + seccomp | 同 Linux |

## 沙箱初始化流程

```
REPL/SDK 啓動
  → main.tsx → init.ts
  → SandboxManager.initialize(sandboxAskCallback)
    → detectWorktreeMainRepoPath()     // 檢測 git worktree，放行主倉庫 .git
    → convertToSandboxRuntimeConfig()  // 構建 SandboxRuntimeConfig
    → BaseSandboxManager.initialize()  // 啓動底層執行時
    → settingsChangeDetector.subscribe() // 訂閱設置變更，動態更新設定
```

`convertToSandboxRuntimeConfig()`（`src/utils/sandbox/sandbox-adapter.ts`）完成從用戶設置到執行時設定的轉換：

1. **網絡規則**：從 `WebFetch(domain:...)` 權限規則提取域名 → `allowedDomains`
2. **檔案系統規則**：從 `Edit(...)` / `Read(...)` 權限規則提取路徑 → `allowWrite` / `denyWrite` / `denyRead`
3. **安全加固**：
   - 自動將專案目錄加入 `allowWrite`
   - 自動將 `settings.json` 路徑加入 `denyWrite`（防止沙箱逃逸）
   - 自動將 `.claude/skills` 加入 `denyWrite`（防止技能注入）
   - 檢測 bare git repo 攻擊向量，對 `HEAD`/`objects`/`refs` 做保護

## `dangerouslyDisableSandbox` 的設計權衡

這個參數的命名本身就傳達了設計意圖——它不是"關閉沙箱"，而是"**危險地禁用沙箱**"。

雙重保險機制：
1. **呼叫側**：模型在 BashTool 的 `inputSchema` 中可以設置 `dangerouslyDisableSandbox: true`
2. **策略側**：管理員可通過 `allowUnsandboxedCommands: false` 完全禁止此參數（企業部署場景）

```typescript
// 即使 AI 請求了 dangerouslyDisableSandbox，策略層仍可覆蓋
if (input.dangerouslyDisableSandbox && 
    SandboxManager.areUnsandboxedCommandsAllowed()) {
  return false  // 只有策略允許時才真正跳過沙箱
}
```

`autoAllowBashIfSandboxed` 進一步補充了這個模型：當啓用時，**在沙箱中的命令自動獲得執行許可**，無需逐條審批。這基於一個信任假設——如果 OS 級沙箱已經限制了命令的能力，那麼應用層的逐條審批就變得多餘。

## 沙箱違規處理

當命令嘗試違反沙箱約束時：

1. 執行時捕獲違規事件（文件/網絡訪問被拒絕）
2. `SandboxManager.annotateStderrWithSandboxFailures()` 在輸出中注入 `<sandbox_violations>` 標籤
3. UI 層通過 `removeSandboxViolationTags()` 清理顯示
4. 違規事件通過 `SandboxViolationStore` 持久化，可用於審計

## 完整執行鏈路示例

以 `npm install` 爲例：

```
1. 用戶在 REPL 中輸入 → Claude 決定呼叫 BashTool
2. BashTool.validateInput() → 通過
3. BashTool.checkPermissions() → 檢查權限規則
   ├── autoAllowBashIfSandboxed = true 且沙箱可用 → 自動允許
   └── 否則 → 彈窗請用戶確認
4. BashTool.call() → runShellCommand()
5. shouldUseSandbox({ command: "npm install" })
   ├── SandboxManager.isSandboxingEnabled() → true
   ├── dangerouslyDisableSandbox → undefined
   └── containsExcludedCommand() → false（除非用戶設定了排除 npm）
   → 結果: true，需要沙箱
6. Shell.exec() → SandboxManager.wrapWithSandbox("npm install")
   ├── macOS: sandbox-exec -p <generated-profile> -- bash -c 'npm install'
   └── Linux: bwrap ... bash -c 'npm install'
7. spawn(wrapped_command) → 子進程在沙箱內執行
8. 執行完成 → SandboxManager.cleanupAfterCommand()
   ├── 清理 bwrap 殘留文件（Linux）
   └── scrubBareGitRepoFiles()（安全清理）
9. 結果返回給 Claude → 展示給用戶
```
