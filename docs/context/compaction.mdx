## 壓縮的觸發時機

上下文壓縮不是單一操作，而是**三層遞進**的策略系統，對應不同的觸發條件和嚴重程度：

| 層級 | 觸發條件 | 實現位置 | 是否需要 API 呼叫 |
|------|---------|---------|:---:|
| **MicroCompact** | 單個工具輸出過長 | `microCompact.ts` | 否 |
| **Session Memory Compact** | 自動壓縮觸發（需 feature flag） | `sessionMemoryCompact.ts` | 否 |
| **傳統 API 摘要** | 手動 `/compact` 或 SM 不可用時的自動回退 | `compact.ts` | 是 |

### 壓縮入口的優先級鏈

源碼路徑：`src/commands/compact/compact.ts`

當用戶執行 `/compact` 或系統觸發自動壓縮時，壓縮命令按以下優先級嘗試：

```typescript
// compact.ts:55-99 — 簡化後的優先級鏈
if (!customInstructions) {
  const sessionMemoryResult = await trySessionMemoryCompaction(messages, ...)
  if (sessionMemoryResult) return sessionMemoryResult      // 優先：SM 壓縮
}

if (reactiveCompact?.isReactiveOnlyMode()) {
  return await compactViaReactive(messages, ...)            // 次選：Reactive 壓縮
}

// 兜底：傳統 API 摘要
const microcompactResult = await microcompactMessages(messages, context)
const messagesForCompact = microcompactResult.messages
// → 呼叫 AI 模型生成摘要
```

注意：SM 壓縮不支援自定義指令（`/compact 聚焦在認證模組`），有自定義指令時直接走傳統路徑。

## 第一層：MicroCompact — 局部壓縮

源碼路徑：`src/services/compact/microCompact.ts`

MicroCompact 不壓縮整個對話，而是**清除舊工具輸出的內容**。它維護一個白名單：

```typescript
const COMPACTABLE_TOOLS = new Set([
  'Read',      // 檔案讀取
  'Bash',      // 命令輸出
  'Grep',      // 搜索結果
  'Glob',      // 檔案列表
  'WebSearch', // 搜索結果
  'WebFetch',  // 網頁內容
  'Edit',      // 編輯輸出
  'Write',     // 寫入輸出
])
```

替換策略：將超過時間窗口的工具輸出內容替換爲 `[Old tool result content cleared]`。這不是簡單的截斷——原始內容仍保留在 JSONL transcript 中，只是不再發送給 API。

MicroCompact 還有一個**時間衰減設定**（`timeBasedMCConfig.ts`）：越舊的工具輸出越容易被清除，最近的優先保留。

### 圖片和文件的特殊處理

```typescript
const IMAGE_MAX_TOKEN_SIZE = 2000
```

圖片 block 如果超過 2000 token 估算值，也會被 MicroCompact 清除。PDF document block 同理。

## 第二層：Session Memory Compact — 無 API 呼叫的壓縮

源碼路徑：`src/services/compact/sessionMemoryCompact.ts`

當 `tengu_session_memory` + `tengu_sm_compact` 兩個 feature flag 啓用時，系統優先使用 Session Memory 進行壓縮——**不需要呼叫摘要模型**，直接使用已經提取好的 Session Memory 作爲對話摘要。

### 保留窗口的計算

```typescript
// sessionMemoryCompact.ts:324-397
export function calculateMessagesToKeepIndex(messages, lastSummarizedIndex) {
  const config = getSessionMemoryCompactConfig()
  // 預設: minTokens=10K, minTextBlockMessages=5, maxTokens=40K

  let startIndex = lastSummarizedIndex + 1
  // 從 lastSummarizedIndex 向前擴展，直到滿足兩個下限或命中上限
  for (let i = startIndex - 1; i >= floor; i--) {
    totalTokens += estimateMessageTokens([msg])
    if (hasTextBlocks(msg)) textBlockMessageCount++
    startIndex = i
    if (totalTokens >= config.maxTokens) break
    if (totalTokens >= config.minTokens && textBlockMessageCount >= config.minTextBlockMessages) break
  }
  return adjustIndexToPreserveAPIInvariants(messages, startIndex)
}
```

這個算法確保壓縮後保留的訊息窗口滿足：
- 至少 10,000 token（有上下文深度）
- 至少 5 條包含文本的訊息（有對話連續性）
- 最多 40,000 token（不會太大又觸發下一次壓縮）

### 工具對完整性保護

`adjustIndexToPreserveAPIInvariants()` 是壓縮中一個**關鍵的正確性保證**：

API 要求每個 `tool_result` 都有對應的 `tool_use`，反之亦然。如果壓縮恰好切在一條 `tool_result` 訊息處，會導致 API 報錯。

```typescript
// sessionMemoryCompact.ts:232-314
// Step 1: 向前掃描，找到所有被保留訊息中 tool_result 引用的 tool_use
// Step 2: 向前掃描，找到與被保留 assistant 訊息共享 message.id 的 thinking block
// 兩種情況都需要將 startIndex 向前移動
```

流式傳輸會將一個 assistant 訊息拆分爲多條存儲記錄（thinking、tool_use 等各有獨立 uuid 但共享 `message.id`），這增加了邊界情況的複雜度。

## 第三層：傳統 API 摘要壓縮

源碼路徑：`src/services/compact/compact.ts`

當 SM 壓縮不可用時，系統回退到傳統方式：呼叫 AI 模型生成對話摘要。

### 壓縮前處理

發送給摘要模型之前，訊息會經過多層預處理：

```typescript
// compact.ts:147-202
const stripped = stripImagesFromMessages(messages)   // 圖片→[image] 文字標記
const stripped2 = stripReinjectedAttachments(stripped) // 移除會被重新注入的附件
```

圖片被替換爲 `[image]` 標記，防止摘要 API 呼叫本身也觸發 prompt-too-long 錯誤。

### 壓縮後的重新注入

壓縮後，系統會從摘要中**重新注入關鍵上下文**：

```typescript
// compact.ts:124-132
export const POST_COMPACT_TOKEN_BUDGET = 50_000          // 總預算
export const POST_COMPACT_MAX_FILES_TO_RESTORE = 5        // 最多恢復 5 個文件
export const POST_COMPACT_MAX_TOKENS_PER_FILE = 5_000     // 每文件 5K token
export const POST_COMPACT_MAX_TOKENS_PER_SKILL = 5_000    // 每技能 5K token
export const POST_COMPACT_SKILLS_TOKEN_BUDGET = 25_000    // 技能總預算 25K
```

這 50K token 的重新注入預算用於：
1. 恢復最近讀取的檔案內容（最多 5 個文件，每個截斷到 5K token）
2. 恢復已激活的技能指令（每個技能截斷到 5K token，總計 25K）
3. 重新注入 CLAUDE.md 內容
4. 恢復 MCP 工具發現結果

## CompactBoundary：壓縮的邊界標記

源碼路徑：`src/utils/messages.ts`（`createCompactBoundaryMessage`）

每次壓縮後，系統在訊息流中插入一條 `SystemCompactBoundaryMessage`：

```typescript
type SystemCompactBoundaryMessage = {
  type: 'system'
  message: {
    type: 'compact_boundary'
    compactMetadata: {
      compactType: 'auto' | 'manual' | 'micro'
      preCompactTokenCount: number
      lastUserMessageUuid: string
      preCompactDiscoveredTools?: string[]
    }
  }
}
```

後續所有操作只處理**最後一條 boundary 之後**的訊息：

```typescript
// messages.ts
export function getMessagesAfterCompactBoundary(messages: Message[]): Message[] {
  const lastBoundary = messages.findLastIndex(m => isCompactBoundaryMessage(m))
  return lastBoundary >= 0 ? messages.slice(lastBoundary + 1) : messages
}
```

### Preserved Segment 註解

boundary 訊息上還附加了 `preservedSegment` 註解，記錄哪些訊息被保留而非壓縮：

```typescript
// compact.ts — annotateBoundaryWithPreservedSegment
boundaryMarker.compactMetadata.preservedSegment = {
  summaryMessageUuid: string
  preservedMessageUuids: string[]
}
```

這在會話恢復時幫助加載器正確重建訊息鏈，避免重複壓縮已保留的訊息。

## PTL 緊急降級：Prompt Too Long

當壓縮後仍然超出 token 限制（`PROMPT_TOO_LONG` 錯誤），系統會進入緊急降級路徑：

1. **Reactive Compact**：`reactiveCompactOnPromptTooLong()` 嘗試更激進的壓縮
2. **截斷重試**：如果 reactive 也失敗，`truncateHeadForPTLRetry()` 直接截斷最早的訊息
3. 放棄並報錯

Reactive Compact 目前在反編譯版本中是 stub（`isReactiveOnlyMode() → false`），表明這是 Anthropic 內部的實驗性功能。

## 壓縮的 Hook 機制

壓縮前後可以執行自定義 Hook：

- **Pre-compact Hook**（`executePreCompactHooks`）：在壓縮前執行，可以注入"必須保留"的標記
- **Post-compact Hook**（`executePostCompactHooks`）：在壓縮後執行，可以驗證關鍵信息是否保留
- **Session Start Hook**（`processSessionStartHooks('compact')`）：SM 壓縮使用此 Hook 恢復 CLAUDE.md 等上下文

Hook 結果以 `HookResultMessage` 的形式附加到壓縮結果中，確保用戶的自定義邏輯在壓縮過程中被尊重。

## Snip Compact（實驗性）

源碼路徑：`src/services/compact/snipCompact.ts`（stub）

Snip Compact 是另一種實驗性壓縮策略，在反編譯版本中爲空殼實現。從 stub 的類型簽名推斷：

```typescript
snipCompactIfNeeded(messages, options?: { force?: boolean }) → {
  messages: Message[]
  executed: boolean
  tokensFreed: number
  boundaryMessage?: Message
}
```

它似乎是一種**更細粒度的訊息級裁剪**（snip = 剪切），可能是對單條訊息的進一步壓縮，而非整個對話。`shouldNudgeForSnips()` 和 `SNIP_NUDGE_TEXT` 暗示它可能會提示用戶觸發。
