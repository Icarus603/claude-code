## 兩種協作模式的架構差異

| 維度 | Coordinator Mode | Agent Swarms |
|------|-----------------|--------------|
| **門控** | `feature('COORDINATOR_MODE')` + `CLAUDE_CODE_COORDINATOR_MODE=1` | 任務系統 V2（預設啓用） |
| **拓撲** | 星型：Coordinator 居中，Worker 外圍 | 網狀：對等 Agent 共享任務列表 |
| **角色** | 明確分工：Coordinator 編排、Worker 執行 | 模糊：每個 Agent 自主認領任務 |
| **通信** | `SendMessage` 定向通信 + `<task-notification>` | 任務檔案系統 + 郵箱廣播 |
| **適用** | 需要集中決策的複雜任務 | 並行度高的獨立子任務 |

兩者不是互斥的——Coordinator Mode 可以在 Swarm 架構之上執行，將 Coordinator 作爲特殊的 Leader Agent。

## Coordinator Mode：星型編排架構

### 激活機制

```typescript
// src/coordinator/coordinatorMode.ts:36
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false  // 外部構建始終 false
}
```

Coordinator Mode 需要雙重門控：構建時 `feature('COORDINATOR_MODE')` 和執行時環境變量。`matchSessionMode()` 在會話恢復時自動同步模式狀態——如果恢復的會話是 coordinator 模式，它會翻轉環境變量以確保一致性。

### Coordinator 的工具集

Coordinator 被剝奪了所有"動手"工具，只保留編排能力：

| 工具 | 用途 |
|------|------|
| **Agent** | 啓動新 Worker（`subagent_type: "worker"`） |
| **SendMessage** | 向已有 Worker 發送後續指令 |
| **TaskStop** | 中途停止走錯方向的 Worker |
| **subscribe_pr_activity** | 訂閱 GitHub PR 事件（review comments、CI 結果） |

Coordinator **不寫程式碼、不讀文件、不執行命令**——它只做三件事：理解需求、分配任務、綜合結果。

### Worker 的工具權限

Worker 的可用工具由 `getCoordinatorUserContext()`（`coordinatorMode.ts:80`）動態注入到 System Prompt：

```typescript
// 簡化模式下：只有 Bash + Read + Edit
const workerTools = isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE')
  ? [BASH_TOOL_NAME, FILE_READ_TOOL_NAME, FILE_EDIT_TOOL_NAME]
  : Array.from(ASYNC_AGENT_ALLOWED_TOOLS)
      .filter(name => !INTERNAL_WORKER_TOOLS.has(name))
```

`INTERNAL_WORKER_TOOLS`（TeamCreate、TeamDelete、SendMessage、SyntheticOutput）被顯式排除——Worker 不能嵌套建立團隊或發送訊息，防止不可控的遞歸。

### Scratchpad：跨 Worker 的共享知識庫

當 `tengu_scratch` feature flag 啓用時，Coordinator 擁有一個 Scratchpad 目錄：

```
Scratchpad 目錄：
  - Workers 可自由讀寫，無需權限審批
  - 用於持久化的跨 Worker 知識
  - 結構由 Coordinator 決定（無固定格式）
```

這是一個關鍵的協作原語——Worker A 的研究結果可以寫入 Scratchpad，Worker B 直接讀取，無需通過 Coordinator 中轉。

### `<task-notification>` 通信協議

Worker 完成後，Coordinator 收到 XML 格式的通知：

```xml
<task-notification>
  <task-id>agent-a1b</task-id>          ← Worker 的 agentId
  <status>completed|failed|killed</status>
  <summary>Agent "Investigate auth bug" completed</summary>
  <result>Found null pointer in src/auth/validate.ts:42...</result>
  <usage>
    <total_tokens>N</total_tokens>
    <tool_uses>N</tool_uses>
    <duration_ms>N</duration_ms>
  </usage>
</task-notification>
```

通知以 `user-role message` 形式送達，Coordinator 通過 `<task-notification>` 標籤區分它和用戶訊息。`<task-id>` 用於 `SendMessage` 的 `to` 參數，實現定向續傳。

### Coordinator 的核心職責：綜合（Synthesis）

Coordinator System Prompt（`coordinatorMode.ts:111-369`，約 260 行）明確要求 Coordinator **不能懶惰地委派理解**：

```
反模式（禁止）：
  "Based on your findings, fix the auth bug"
  → 把理解的責任推給了 Worker

正確做法：
  "Fix the null pointer in src/auth/validate.ts:42.
   The user field on Session (src/auth/types.ts:15) is
   undefined when sessions expire but the token remains cached.
   Add a null check before user.id access."
  → Coordinator 自己理解了問題，給出精確指令
```

這是 Coordinator Mode 最核心的設計約束：Coordinator 必須先理解，再分配。

## Agent Swarms：蜂羣式協作

Swarm 模式基於任務系統 V2（詳見[任務管理](../tools/task-management.mdx)），核心機制是**共享任務列表 + 競爭認領**：

### 團隊初始化

```
Leader 建立團隊（TeamCreateTool）
  ↓
設置 teamName → setLeaderTeamName()
  ↓
所有 teammate 自動獲得相同的 taskListId
  ↓
teammate 啓動時：
  1. CLAUDE_CODE_TASK_LIST_ID 環境變量（顯式覆蓋）
  2. teammate 上下文的 teamName（共享 leader 的任務列表）
  3. CLAUDE_CODE_TEAM_NAME 環境變量
  4. leader 設置的 teamName
  5. getSessionId()（兜底）
```

多級優先級確保了 Leader 和所有 Teammate 指向同一個任務列表，無需額外協調。

### 任務認領與競爭

`claimTask()` 是 Swarm 的核心併發原語：

```
Teammate A 呼叫 TaskList → 發現 task #3 是 pending
Teammate B 同時發現 task #3 是 pending
  ↓
兩者同時嘗試 TaskUpdate(task #3, {status: "in_progress"})
  ↓
檔案鎖 + 高水位標記保證原子性：
  - 第一個寫入者獲得 owner 鎖定
  - 第二個寫入者收到 already_claimed 錯誤
  ↓
獲得任務的 teammate 執行工作
  ↓
完成後 TaskUpdate(task #3, {status: "completed"})
  → 依賴此任務的其他任務自動解鎖
  → tool_result 提示 "Call TaskList to find your next task"
```

### Teammate 的生命週期管理

```
Teammate 異常退出
  ↓
unassignTeammateTasks()
  → 掃描任務列表，找到 owner === teammateName 的未完成任務
  → 重置爲 pending + owner=undefined
  ↓
Leader 通過 mailbox 收到通知
  → 重新分配或建立新 Teammate
```

## 任務類型全景

支撐多 Agent 協作的是 7 種任務類型（`src/tasks/types.ts`）：

| 任務類型 | 執行位置 | 狀態管理 | 適用場景 |
|----------|---------|---------|---------|
| **LocalAgentTask** | 本地子進程 | `LocalAgentTaskState` | 標準子 Agent 任務 |
| **LocalShellTask** | 本地 shell | `LocalShellTaskState` | 後臺 shell 命令 |
| **InProcessTeammateTask** | 同進程內 | `InProcessTeammateTaskState` | 輕量級進程內隊友 |
| **RemoteAgentTask** | 遠程服務器 | `RemoteAgentTaskState` | 分佈式 Agent（CCR） |
| **DreamTask** | 後臺靜默 | `DreamTaskState` | 後臺自主整理記憶 |
| **LocalWorkflowTask** | 本地 | `LocalWorkflowTaskState` | 工作流編排 |
| **MonitorMcpTask** | 本地 | `MonitorMcpTaskState` | MCP 監控任務 |

`InProcessTeammateTask` 與 `LocalAgentTask` 的關鍵差異：前者共享進程的內存空間和基礎設施狀態（如 MCP 連接池），但有獨立的對話上下文和工具權限；後者是完全隔離的子進程，啓動開銷更大但更安全。

## Coordinator vs Swarm 的選擇

| 場景 | 推薦模式 | 原因 |
|------|---------|------|
| "重構認證系統，需要多模組協調" | Coordinator | 需要集中決策，Worker 間有依賴 |
| "修復 10 個獨立的 lint 警告" | Swarm | 任務獨立，可完全並行 |
| "研究方案 A 和方案 B，然後選一個實現" | Coordinator | 先並行研究，再集中決策 |
| "在大倉庫中搜索所有 TODO 並分類" | Swarm | 無依賴，各自領任務即可 |
