## 五層架構

Claude Code 從上到下分爲五個層次，每一層職責清晰、邊界分明：

<Frame caption="Claude Code 五層架構">
  <img src="/docs/images/architecture-layers.png" alt="Claude Code 五層架構圖" />
</Frame>

| 層次 | 職責 | 入口源碼 | 關鍵詞 |
|------|------|---------|--------|
| **交互層** | 終端 UI、用戶輸入、訊息展示 | `src/screens/REPL.tsx` | React/Ink、PromptInput |
| **編排層** | 多輪對話、會話持久化、成本追蹤 | `src/QueryEngine.ts` | QueryEngine、transcript |
| **核心循環層** | 單輪：發請求 → 拿響應 → 執行工具 → 循環 | `src/query.ts` | Agentic Loop、State |
| **工具層** | AI 的"雙手"——讀寫文件、執行命令 | `src/tools.ts` → `src/Tool.ts` | Tool 介面、MCP |
| **通信層** | 與 Claude API 的流式通信 | `src/services/api/claude.ts` | Streaming、Provider |

## 一條主資料流的源碼追蹤

<Frame caption="核心資料流">
  <img src="/docs/images/data-flow.png" alt="Claude Code 核心資料流" />
</Frame>

整個系統的運轉可以濃縮爲一條核心資料流，以下是每一步對應的源碼路徑：

### 1. 用戶輸入 → REPL

`src/screens/REPL.tsx` 是基於 React/Ink 的終端 UI 組件。用戶輸入經 `processUserInput()`（`src/utils/processUserInput/processUserInput.ts`）處理，支援斜槓命令、文件附件、圖片等。

### 2. QueryEngine 編排

`src/QueryEngine.ts` 是 REPL 與 `query()` 之間的中間層，管理：
- **會話狀態**：訊息數組、工具權限上下文（`ToolPermissionContext`）、檔案歷史快照
- **成本追蹤**：`accumulateUsage()` / `getTotalCost()` 累計 token 用量
- **Transcript 持久化**：`recordTranscript()` 將對話序列化到磁盤，支援 `--resume`
- **檔案歷史**：`fileHistoryMakeSnapshot()` 在修改前建立快照，支援 undo

關鍵方法：`queryEngine.query()` 構造 `QueryParams`，呼叫 `query()` 異步生成器。

### 3. Agentic Loop（`src/query.ts`）

`query()` 是一個 `AsyncGenerator`，`while(true)` 循環的每次迭代包含：

```
① 上下文預處理管道：
   applyToolResultBudget → snipCompact → microcompact → contextCollapse → autocompact

② 流式 API 呼叫：
   deps.callModel() → AsyncGenerator<StreamEvent | Message>
   收集 assistantMessages[]、toolUseBlocks[]

③ 工具執行：
   StreamingToolExecutor（並行） 或 runTools（串行）
   → toolResults[]

④ 終止/繼續判定：
   needsFollowUp ? continue : return { reason }
```

完整的狀態機通過 `State` 類型（`src/query.ts:204`）在迭代間傳遞，包含 10 個字段（messages、autoCompactTracking、maxOutputTokensRecoveryCount 等）。

### 4. 工具層（`src/tools.ts` → `src/Tool.ts`）

`getAllBaseTools()`（`src/tools.ts:191`）組裝 50+ 工具列表，經過 `filterToolsByDenyRules()` 權限過濾後傳給 API。

每個工具實現 `Tool<Input, Output, Progress>` 介面（`src/Tool.ts:362`），核心方法鏈：
```
validateInput() → canUseTool()（UI 層）→ checkPermissions() → call() → ToolResult
```

### 5. 通信層（`src/services/api/claude.ts`）

API 客戶端支援 4 種 Provider：
- **Anthropic Direct**：預設
- **AWS Bedrock**：`ANTHROPIC_BEDROCK_BASE_URL`
- **Google Vertex**：`ANTHROPIC_VERTEX_PROJECT_ID`
- **Azure**：通過自定義 base URL

`deps.callModel()` 發起流式請求，返回 `BetaRawMessageStreamEvent` 事件流。支援 Prompt Cache（`cache_control`）、thinking blocks、multi-turn tool use。

## 四個核心設計原則

<AccordionGroup>
  <Accordion title="流式優先 (Streaming-first)">
    所有 API 通信都是流式的——`deps.callModel()` 返回 AsyncGenerator，用戶看到 AI "逐字打出"回答。StreamingToolExecutor 在流式過程中就開始並行執行工具，不等流結束。模型降級（Fallback）時，已收集的 assistantMessages 被標記爲 tombstone 並清空，重試整個流式請求。
  </Accordion>
  <Accordion title="工具即能力 (Tool as Capability)">
    每個工具是 `Tool<Input, Output, Progress>` 結構化類型，通過 `buildTool()` 工廠建立。`getTools()` 在每次 API 呼叫時組裝（非全局快取），因爲 `isEnabled()` 可能隨執行時狀態變化。MCP 工具通過 `mcpInfo` 字段標記來源，支援 server 級別的 blanket deny。
  </Accordion>
  <Accordion title="權限即邊界 (Permission as Boundary)">
    每次工具呼叫經過 `validateInput() → checkPermissions()` 雙重檢查。權限規則從 5 個來源匯聚（session → project → user → managed → default），支援工具名、命令模式、路徑前綴等匹配方式。Plan Mode 通過 `prepareContextForPlanMode()` 切換爲只讀模式，退出時自動恢復。
  </Accordion>
  <Accordion title="上下文即記憶 (Context as Memory)">
    System Prompt 由 `fetchSystemPromptParts()` 動態組裝，包含 CLAUDE.md、git 狀態、日期、MCP 服務器列表。Auto-compact 在每輪迭代前評估 token 閾值，超出時觸發壓縮。壓縮後的摘要通過 `buildPostCompactMessages()` 替換原始訊息，`taskBudgetRemaining` 跨壓縮邊界累計。
  </Accordion>
</AccordionGroup>

## 入口與引導

| 入口 | 文件 | 說明 |
|------|------|------|
| CLI 啓動 | `src/entrypoints/cli.tsx` | 注入 `feature()` polyfill（始終返回 false）、MACRO 全局變量 |
| 命令定義 | `src/main.tsx` | Commander.js 解析參數，初始化 auth/analytics/policy |
| 一次性初始化 | `src/entrypoints/init.ts` | 遙測設定、信任對話框 |
| 管道模式 | `src/main.tsx` `-p` flag | `echo "say hello" \| bun run dev -p` |
