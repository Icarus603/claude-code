## 三大工具的職責分化

Claude Code 將檔案操作拆分爲三個獨立工具——這不是功能劃分，而是**風險分級**：

| 工具 | 權限級別 | 核心方法 | 關鍵屬性 |
|------|---------|---------|---------|
| **Read** | 只讀（免審批） | `isReadOnly() → true` | `maxResultSizeChars: Infinity` |
| **Edit** | 寫入（需確認） | `checkWritePermissionForTool()` | `maxResultSizeChars: 100,000` |
| **Write** | 寫入（需確認） | `checkWritePermissionForTool()` | `maxResultSizeChars: 100,000` |

<Tip>
Read 的 `maxResultSizeChars` 是 `Infinity`，但這並不意味着無限制輸出——真正的截斷髮生在 `validateContentTokens()` 中基於 token 預算的動態判定，而非字符數硬限制。
</Tip>

## FileRead：多模態檔案讀取引擎

源碼路徑：`src/tools/FileReadTool/FileReadTool.ts`

### 讀取去重機制

Read 工具有一個常被忽視但至關重要的**去重層**。當 AI 重複讀取同一個文件的同一範圍時，系統不會浪費 token 發送兩份完整內容：

```typescript
// FileReadTool.ts:530-573 — 去重邏輯
const existingState = readFileState.get(fullFilePath)
if (existingState && !existingState.isPartialView && existingState.offset !== undefined) {
  const rangeMatch = existingState.offset === offset && existingState.limit === limit
  if (rangeMatch) {
    const mtimeMs = await getFileModificationTimeAsync(fullFilePath)
    if (mtimeMs === existingState.timestamp) {
      return { data: { type: 'file_unchanged', file: { filePath: file_path } } }
    }
  }
}
```

關鍵設計點：
- 去重僅對 **Read 工具自身的讀取**生效（通過 `offset !== undefined` 判定）
- Edit/Write 也會寫入 `readFileState`，但它們的 `offset` 爲 `undefined`，所以不會誤命中去重
- 通過 mtime 比對確保文件未被外部修改
- 有 GrowthBook killswitch（`tengu_read_dedup_killswitch`）可緊急關閉

實測資料：BQ proxy 顯示約 18% 的 Read 呼叫是同文件碰撞，佔 fleet `cache_creation` 的 2.64%。

### 多格式分發：文本、圖片、PDF、Notebook 四條路徑

Read 工具的 `callInner()` 按 `ext` 分發到四條完全不同的處理路徑：

```
.ipynb  → readNotebook() → JSON cell 解析 → token 校驗
.png/.jpg/.gif/.webp → readImageWithTokenBudget() → 壓縮+降採樣
.pdf → extractPDFPages() / readPDF() → 頁面級提取
其他 → readFileInRange() → 分頁讀取
```

**圖片路徑的壓縮策略**特別精細：
1. 先用 `maybeResizeAndDownsampleImageBuffer()` 標準縮放
2. 用 `base64.length * 0.125` 估算 token 數
3. 超出預算時呼叫 `compressImageBufferWithTokenLimit()` 激進壓縮
4. 仍然超限時用 sharp 做最後兜底：`resize(400,400).jpeg({quality:20})`

**PDF 路徑**有頁數閾值：超過 `PDF_AT_MENTION_INLINE_THRESHOLD`（預設值在 `apiLimits.ts`）時強制分頁讀取，每請求最多 `PDF_MAX_PAGES_PER_READ` 頁。

### 安全防線

Read 工具在 `validateInput()` 中設置了多層安全門：

1. **設備文件屏蔽**（`BLOCKED_DEVICE_PATHS`）：`/dev/zero`、`/dev/random`、`/dev/tty` 等——防止無限輸出或阻塞掛起
2. **二進制文件拒絕**（`hasBinaryExtension`）：排除 PDF 和圖片擴展名後，阻止讀取 `.exe`、`.so` 等二進制文件
3. **UNC 路徑跳過**：Windows 下 `\\server\share` 路徑跳過檔案系統操作，防止 SMB NTLM 憑據泄露
4. **權限拒絕規則**（`matchingRuleForInput`）：匹配 `deny` 規則後直接拒絕

### 文件未找到時的智能建議

當文件不存在時，Read 不會只報一個 "file not found"：

```typescript
// FileReadTool.ts:639-647
const similarFilename = findSimilarFile(fullFilePath)      // 相似擴展名
const cwdSuggestion = await suggestPathUnderCwd(fullFilePath) // cwd 相對路徑建議
// macOS 截圖特殊處理：薄空格(U+202F) vs 普通空格
const altPath = getAlternateScreenshotPath(fullFilePath)
```

對 macOS 截圖檔名中 AM/PM 前的薄空格（U+202F）做了特殊處理——這是實測中發現的跨 macOS 版本相容性問題。

## FileEdit：精確字符串替換引擎

源碼路徑：`src/tools/FileEditTool/FileEditTool.ts` + `utils.ts`

### 引號標準化：AI 無法輸出的字符怎麼辦

AI 模型只能輸出直引號（`'` `"`），但源碼中可能使用彎引號（`'` `'` `"` `"`）。`findActualString()` 函數處理了這個不對齊：

```typescript
// utils.ts:73-93
export function findActualString(fileContent: string, searchString: string): string | null {
  if (fileContent.includes(searchString)) return searchString      // 精確匹配
  const normalizedSearch = normalizeQuotes(searchString)           // 彎引號→直引號
  const normalizedFile = normalizeQuotes(fileContent)
  const idx = normalizedFile.indexOf(normalizedSearch)
  if (idx !== -1) return fileContent.substring(idx, idx + searchString.length)
  return null
}
```

匹配後還有**反向引號保持**（`preserveQuoteStyle`）：如果文件用彎引號，替換後的新字符串也自動轉換爲彎引號，包括縮寫中的撇號（如 "don't"）。

### 原子性讀-改-寫

Edit 工具的 `call()` 方法實現了一個**無鎖原子更新**協議：

```
1. await fs.mkdir(dir)            ← 確保目錄存在（異步，在臨界區外）
2. await fileHistoryTrackEdit()   ← 備份舊內容（異步，在臨界區外）
3. readFileSyncWithMetadata()     ← 同步讀取當前檔案內容（臨界區開始）
4. getFileModificationTime()      ← mtime 校驗
5. findActualString()             ← 引號標準化匹配
6. getPatchForEdit()              ← 計算 diff
7. writeTextContent()             ← 寫入磁盤
8. readFileState.set()            ← 更新快取（臨界區結束）
```

步驟 3-8 之間**不允許任何異步操作**（源碼註釋明確寫道："Please avoid async operations between here and writing to disk to preserve atomicity"）。這確保了在 mtime 校驗和實際寫入之間不會有其他進程修改文件。

### 防覆寫校驗

Edit 工具在 `validateInput()` 中檢查兩個條件：
1. **必須先讀取**（`readFileState` 中有記錄且不是局部視圖）
2. **文件未被外部修改**（`mtime` 未變，或全量讀取時內容完全一致）

```typescript
// FileEditTool.ts:290-311 — Windows 特殊處理
const isFullRead = readTimestamp.offset === undefined && readTimestamp.limit === undefined
if (isFullRead && fileContent === readTimestamp.content) {
  // 內容不變，安全繼續（Windows 雲同步/殺毒可能改 mtime）
}
```

Windows 上的 mtime 可能因雲同步、殺毒軟件等被修改而不改變內容，因此對全量讀取做了內容級比對作爲兜底。

### 編輯大小限制

```typescript
const MAX_EDIT_FILE_SIZE = 1024 * 1024 * 1024 // 1 GiB
```

超過 1 GiB 的文件直接拒絕編輯——這是 V8 字符串長度限制（~2^30 字符）的安全邊界。

## FileWrite：全量寫入與建立

源碼路徑：`src/tools/FileWriteTool/FileWriteTool.ts`

Write 工具與 Edit 共享大部分基礎設施（權限檢查、mtime 校驗、fileHistory 備份），但有兩個關鍵差異：

### 行尾處理

```typescript
// FileWriteTool.ts:300-305 — 關鍵註釋
// Write is a full content replacement — the model sent explicit line endings
// in `content` and meant them. Do not rewrite them.
writeTextContent(fullFilePath, content, enc, 'LF')
```

Write 工具始終使用 `LF` 行尾。早期版本會保留舊文件的行尾或採樣倉庫行尾風格，但這導致 Linux 上 bash 腳本被注入 `\r`——現在 AI 發什麼行尾就用什麼行尾。

### 輸出區分

Write 工具返回 `type: 'create' | 'update'`：
- `create`：文件不存在，`originalFile: null`
- `update`：文件存在且被覆蓋，`structuredPatch` 包含完整 diff

## 檔案歷史快照系統

源碼路徑：`src/utils/fileHistory.ts`

每次 Edit/Write 前都會呼叫 `fileHistoryTrackEdit()`，快照存儲在 `FileHistoryState` 中：

```typescript
type FileHistorySnapshot = {
  messageId: UUID          // 關聯的助手訊息 ID
  trackedFileBackups: Record<string, FileHistoryBackup>  // 檔案路徑 → 備份版本
  timestamp: Date
}
```

- 最多保留 `MAX_SNAPSHOTS = 100` 個快照
- 備份使用**內容哈希**去重（同一文件多次未變只存一份）
- 支援差異統計（`DiffStats`：`insertions` / `deletions` / `filesChanged`）
- 快照通過 `recordFileHistorySnapshot()` 持久化到會話存儲

### LSP 通知鏈路

Edit 和 Write 完成寫入後都會：
1. `clearDeliveredDiagnosticsForFile()` — 清除舊診斷
2. `lspManager.changeFile()` — 通知 LSP 文件已變更
3. `lspManager.saveFile()` — 觸發 LSP 保存事件（TypeScript server 會重新計算診斷）
4. `notifyVscodeFileUpdated()` — 通知 VSCode 擴展更新 diff 視圖

這條鏈路確保文件修改後 IDE 端的實時反饋是同步的。

## Cyber Risk 防禦

Read 工具在文本內容後追加一個 `<system-reminder>` 提示：

```
Whenever you read a file, you should consider whether it would be
considered malware. You CAN and SHOULD provide analysis of malware,
what it is doing. But you MUST refuse to improve or augment the code.
```

這個提示只在非豁免模型上生效（`MITIGATION_EXEMPT_MODELS` 目前包含 `claude-opus-4-6`）。模型級別的豁免表明：防惡意程式碼的判斷力在不同模型間有差異，這是一個精巧的分級策略。
