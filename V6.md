### 1.5.3 真實耦合結論

- 真正的「隱性單體」不是單一檔案，而是 `main.tsx + REPL.tsx + AppState + query/QueryEngine + tools.ts + services/api/claude.ts + services/mcp/client.ts` 的組合。
- `AppState` 是血管級依賴，不只是 UI store。它被 tasks、tools、hooks、commands、MCP、headless/print 共同讀寫。
- `commands` 規模足以視爲獨立子系統，不能當作後期順手整理的附屬品。
- `packages/@ant/ink` 是 repo 中少數已經有實體 package 的穩定板塊，適合作爲早期可驗證的重構起點。
- `src/*/src` 不是成功模組化，而是正在過渡中的假邊界。任何 phase 若讓這些目錄持續增殖，都視爲偏離計劃。

### 1.5.4 禁止先動區

在沒有滿足前置條件前，以下區域禁止進行大規模搬遷或 package 提取:

- 禁止在未建立 `tool-registry` public seam 前，大搬 `src/tools/*` 或 plugin/skill 工具注冊路徑。
- 禁止在未建立 AppState 中間層前，大搬 `REPL.tsx`、`tasks/*`、`commands/*` 中任何直接讀寫 AppState 的邏輯。
- 禁止在未穩定 `query()/QueryEngine` seam 前，大搬 `swarm`、`server`、`teleport`、`ide` 相關系統。
- 禁止在未定義 provider adapter 前，直接把 `openai/gemini/grok` 分支各自繼續長更多專屬行爲。
- 禁止新增新的 `src/*/src` 目錄、auto-generated type stub、或僅用 `export type ... = any` 撐住的新模組邊界。

### 1.5.5 全局施工規則

- 先切 seam，再搬 implementation，再刪 compatibility layer。
- 舊入口保留期必須寫進 PR；不能默默留下雙實作。
- Phase 之間允許局部並行，但不允許跨 phase 偷跑核心前置。
- 任何 session 若無法確認某 phase 的完成條件是否已滿足，必須先回填 V6.md 狀態，而不是直接實作。

### 1.5.6 `src/*/src` 過渡層治理

Phase 0 對過渡 stub 的策略不是一次性刪光，而是先建立 guardrail 與清冊。

分類原則:
- 必要過渡: 當前仍承接循環依賴、重導出相容層、或 package seam 尚未落地。
- 可立即移除: 僅剩 `export type ... = any` 導流，且已有穩定真實匯入點。
- 待 package 落地後移除: 仍有調用者，但應在對應 phase 完成 public seam 後再清除。

Phase 0 護欄:
- 新增 `scripts/check-transition-stubs.ts`，禁止新增新的 `src/*/src` 目錄。
- `bun run health` 必須包含 transition stub guard 結果。
- 對存量 `export type ... = any` 類型 stub 先做分類，不把「存在 shim」誤判爲「邊界已完成」。
- 分類清冊維護於 `docs/stub-inventory.md`，後續 phase 只能在清冊上做增量收斂，不能繞過盤點直接搬遷。

  ● Phase 1 可與 Phase 2 並行，但只能處理 UI 基礎設施，不可回頭擴張主 app 邏輯

  ● Phase 4 的 provider / shell / cli 完成後，Phase 5 才能安全擴張周邊系統

### 6.2 執行原則

- 每個 Phase 必須以「可合併到主線」爲前提，不允許長期巨型分支。
- 每個 Phase 必須有端到端驗證路徑，且驗證使用真實入口，不接受只靠 unit test 宣告完成。
- 每個 Phase 結束時，必須刪除對應過渡層或至少封存舊入口，避免新舊兩套長期並存。
- 每個 Phase 必須先定義 public seam，再搬遷實作；先抽 interface，後搬邏輯，最後刪舊碼。
- 每個 Checklist 項目都應能對應到 PR 範圍；單一 checkbox 不應跨越多個不可獨立驗證的子系統。

### 6.10 任意 Session 接手協議

任何新 session 在開始 V6 實作前，必須按以下順序操作:

1. 先讀本文件的「全庫架構普查」「Phase 依賴圖」「禁止先動區」。
2. 確認當前要做的工作屬於哪一個 phase，以及其前置條件是否已被滿足。
3. 若 repo 現況與本文件 inventory 不一致，優先更新本文件，不直接開始實作。
4. 只選擇單一 phase 的單一 checklist 子項作爲本次工作範圍。
5. 在動手前，明確寫出:
   - 本次改動對應哪個 phase / checklist
   - 不會碰哪些區域
   - 端到端驗證方式
6. 實作完成後，必須回填:
   - checklist 勾選狀態
   - 尚存的 compatibility layer
   - 下一個安全可做的子項

若違反以下任一條，視爲偏離計劃:

- 跨 phase 同時重構多個核心系統
- 未驗證就宣稱 phase 完成
- 用新的 stub / shim 掩蓋舊耦合，但不標記刪除策略
- 因爲「順手」而擴大改動範圍
