## Tool vs Skill：本質差異

| | Tool | Skill |
|---|---|---|
| 粒度 | 單個原子操作（讀文件、執行命令） | 一套完整的工作流（程式碼審查、建立 PR） |
| 觸發方式 | AI 自主選擇 | 用戶 `/skill-name` 或 AI 通過 `SkillTool` 自動匹配 |
| 本質 | TypeScript 執行邏輯 | **Prompt + 權限設定**的聲明式封裝 |
| 註冊位置 | `src/tools.ts` → `getTools()` | `src/commands.ts` → `getCommands()` |
| 執行器 | 各 Tool 的 `call()` 方法 | `SkillTool.call()` → 兩條分支（inline / fork） |

Skill 的核心洞見：**複雜任務的關鍵不在程式碼邏輯，而在 Prompt 質量**。一個程式碼審查 Skill 不需要審查引擎，只需告訴 AI "審查什麼、按什麼順序、輸出什麼格式"——Skill 把這種"經驗"封裝爲可複用的 Markdown。

## Skill 的五個來源與加載鏈路

### 1. 內置命令（Built-in Commands）

硬編碼在 `src/commands.ts:258` 的 `COMMANDS` memoize 數組中，包含 70+ 條命令（`/commit`、`/review`、`/compact` 等）。這些是 TypeScript 模組而非 Markdown，但實現了相同的 `Command` 介面（`src/types/command.ts`）。

### 2. Bundled Skills（編譯時打包）

通過 `registerBundledSkill()`（`src/skills/bundledSkills.ts:53`）在模組初始化時註冊。關鍵特性：

- **延遲文件提取**：如果 Skill 聲明瞭 `files`（參考文件），首次呼叫時才解壓到臨時目錄（`getBundledSkillExtractDir()`），使用 `O_NOFOLLOW | O_EXCL` 防止符號連結攻擊（`safeWriteFile`，第 186 行）
- **閉包級 memoize**：併發呼叫共享同一個 extraction promise，避免競態寫入
- 來源標記爲 `source: 'bundled'`，在 Prompt 預算中享有**不可截斷**的特權

### 3. 磁盤 Skills（`.claude/skills/`）

由 `loadSkillsFromSkillsDir()`（`src/skills/loadSkillsDir.ts:407`）加載，這是最重要的加載路徑：

```
管理策略: $MANAGED_DIR/.claude/skills/     (policySettings)
用戶全局: ~/.claude/skills/                 (userSettings)
專案級:   .claude/skills/                   (projectSettings, 向上遍歷至 home)
附加目錄: --add-dir 指定的路徑下 .claude/skills/
```

**加載協議**：只識別 `skill-name/SKILL.md` 目錄格式，不再支援單檔 `.md`。加載流程：

1. `readdir` 掃描目錄 → 僅保留 `isDirectory()` 或 `isSymbolicLink()` 的條目
2. 在每個子目錄中查找 `SKILL.md`，未找到則跳過
3. `parseFrontmatter()` 解析 YAML 頭部，提取 `whenToUse`、`allowedTools`、`context` 等字段
4. `parseSkillFrontmatterFields()`（第 185 行）統一解析 17 個 frontmatter 字段
5. `createSkillCommand()`（第 270 行）構造 `Command` 對象

**去重機制**：使用 `realpath()` 解析符號連結獲得規範路徑（`getFileIdentity`，第 118 行），避免通過符號連結或重疊父目錄導致的重複加載。

### 4. MCP Skills（動態發現）

通過 `registerMCPSkillBuilders()` 註冊構建器，MCP Server 的 prompt 被 `mcpSkillBuilders.ts` 轉換爲 `Command` 對象。標記爲 `loadedFrom: 'mcp'`。

**安全邊界**：MCP Skills 的 Prompt 內容**禁止執行內聯 shell 命令**（`loadSkillsDir.ts:374` 的 `loadedFrom !== 'mcp'` 守衛），因爲遠程內容不可信。

### 5. Legacy Commands（`/commands/` 目錄）

向後相容的舊格式，由 `loadSkillsFromCommandsDir()`（第 566 行）加載。同時支援 `SKILL.md` 目錄格式和單 `.md` 檔案格式。

## Frontmatter 字段全景

一個 `SKILL.md` 的完整 frontmatter（`parseSkillFrontmatterFields`，第 185 行）：

```yaml
---
name: code-review                    # 顯示名稱（覆蓋目錄名）
description: 系統性程式碼審查           # 描述（或從 Markdown 首段提取）
when_to_use: "用戶說審查程式碼、找 bug"  # AI 自動匹配依據
allowed-tools:                       # 工具白名單
  - Read
  - Grep
  - Glob
argument-hint: "<file-or-directory>" # 參數提示
arguments: [path]                    # 聲明式參數名（用於 $ARGUMENTS 替換）
model: opus                          # 模型覆蓋
effort: high                         # 努力級別
context: fork                        # 執行模式：inline（預設）| fork
agent: code-reviewer                 # 指定 Agent 定義文件
user-invocable: true                 # 用戶是否可 /呼叫
disable-model-invocation: false      # 禁止 AI 自主呼叫
version: "1.0"                       # 版本號
paths:                               # 條件激活的檔案路徑模式
  - "src/**/*.ts"
hooks:                               # Hook 設定
  PreToolUse:
    - command: ["echo", "checking"]
shell: ["bash"]                      # Shell 執行環境
---
```

解析後有 17 個字段被提取，其中 `allowedTools`、`model`、`effort` 在執行時動態修改 `toolPermissionContext`。

## 兩條執行路徑：Inline vs Fork

SkillTool（`src/tools/SkillTool/SkillTool.ts:332`）在 `call()` 中根據 `command.context` 分流：

### Inline 模式（預設）

Skill 的 Prompt 內容被注入爲 **UserMessage**，在主對話流中繼續執行：

1. `processPromptSlashCommand()` 處理參數替換（`$ARGUMENTS`）和 shell 命令展開（`` !`...` ``）
2. `${CLAUDE_SKILL_DIR}` 被替換爲 Skill 所在目錄的絕對路徑
3. `${CLAUDE_SESSION_ID}` 被替換爲當前會話 ID
4. 返回 `newMessages`（注入到對話流）+ `contextModifier`（修改權限上下文）

`contextModifier`（第 776 行）做了三件事：
- **工具白名單注入**：將 `allowedTools` 合併到 `alwaysAllowRules.command`
- **模型切換**：`resolveSkillModelOverride()` 處理模型覆蓋，保留 `[1m]` 後綴以避免 200K 窗口截斷
- **努力級別覆蓋**：修改 `effortValue`

### Fork 模式（`context: fork`）

Skill 在**獨立子 Agent** 中執行（`executeForkedSkill`，第 122 行）：

1. `prepareForkedCommandContext()` 構建隔離的 Agent 定義和 Prompt
2. `runAgent()` 啓動子 Agent 循環，擁有獨立的 token 預算
3. 通過 `onProgress` 回調報告工具使用進度
4. 結果通過 `extractResultText()` 提取，子 Agent 的全部訊息在提取後被釋放（`agentMessages.length = 0`）
5. 最終通過 `clearInvokedSkillsForAgent()` 清理狀態

Fork 模式適用於需要強隔離的場景（如長時間執行的審查任務），避免污染主對話的上下文。

## 權限模型：Safe Properties 白名單

`checkPermissions()`（第 433 行）實現了一個四層權限檢查：

```
1. Deny 規則匹配（支援精確匹配和 prefix:* 通配符）
   ↓ 未命中
2. 官方市場 Skill 自動放行（plugin + isOfficialMarketplaceName）
   ↓ 未命中
3. Allow 規則匹配
   ↓ 未命中
4. Safe Properties 白名單檢查（skillHasOnlySafeProperties，第 911 行）
   ↓ 有非安全屬性
5. Ask 用戶確認（附帶精確匹配和前綴匹配兩條建議規則）
```

**Safe Properties**（`SAFE_SKILL_PROPERTIES`，第 876 行）是一個包含 28 個屬性名的白名單。任何不在白名單中的**有意義的屬性值**（排除 `undefined`、`null`、空數組、空對象）都會觸發權限請求。這是**正向安全**設計——未來新增的屬性預設需要權限。

## Prompt 預算：1% 上下文窗口的截斷策略

Skill 列表注入 System Prompt 時有嚴格的字符預算（`prompt.ts`）：

- **預算計算**：`contextWindowTokens × 4 chars/token × 1%`（約 8000 字符）
- **單條上限**：`MAX_LISTING_DESC_CHARS = 250` 字符（超出截斷爲 `…`）
- **Bundled Skills 不可截斷**：它們始終保留完整描述，預算不足時只截斷非 bundled 的
- **降級策略**：
  1. 嘗試完整描述 → 超預算？
  2. Bundled 保留完整，非 bundled 均分剩餘預算 → 每條描述低於 20 字符？
  3. 非 bundled 僅保留名稱

`formatCommandsWithinBudget()`（`prompt.ts:70`）實現了這個三級降級。

## 動態發現與條件激活

### 基於檔案路徑的動態發現

`discoverSkillDirsForPaths()`（`loadSkillsDir.ts:861`）在檔案操作時觸發：

1. 從被操作的檔案路徑開始，**向上遍歷**至 CWD（不包含 CWD 本身）
2. 在每層查找 `.claude/skills/` 目錄
3. 使用 `realpath` 去重，`git check-ignore` 過濾 gitignored 目錄
4. 按路徑深度排序（**深層優先**），更接近文件的 Skill 優先級更高

### 條件激活（paths frontmatter）

帶有 `paths` 模式的 Skill 在加載時不會立即可用，而是存入 `conditionalSkills` Map。當被操作的檔案路徑匹配某個 Skill 的 paths 模式時（使用 `ignore` 庫做 gitignore 風格匹配），該 Skill 才被**激活**——從 `conditionalSkills` 移入 `dynamicSkills`。

這意味着一個只在 `*.test.ts` 上激活的測試 Skill，平時完全不可見，只有當 AI 讀取或編輯測試文件時纔會出現。

## 使用頻率排名

`recordSkillUsage()`（`skillUsageTracking.ts`）使用指數衰減算法計算 Skill 排名分數：

```
score = usageCount × max(0.5^(daysSinceUse / 7), 0.1)
```

- **7 天半衰期**：一週前的使用權重減半
- **最低 0.1 保底**：避免老但高頻使用的 Skill 完全沉底
- **60 秒去抖**：同一 Skill 在 1 分鐘內的多次呼叫只計一次，減少檔案 I/O

排名資料持久化在全局設定的 `skillUsage` 字段中。

## 遠程技能加載（Experimental）

通過 `EXPERIMENTAL_SKILL_SEARCH` feature flag 控制，支援從遠程（AKI/GCS/S3）加載 `_canonical_<slug>` 格式的 Skill：

1. `validateInput()` 中 `stripCanonicalPrefix()` 攔截 canonical 名稱
2. `executeRemoteSkill()`（第 970 行）從遠程 URL 加載 SKILL.md
3. 支援 `gs://`、`https://`、`s3://` 等 URL 協議
4. 內容經過 frontmatter 剝離、`${CLAUDE_SKILL_DIR}` 替換後直接注入
5. 通過 `addInvokedSkill()` 註冊到 compaction 保留狀態，確保壓縮後仍可恢復
6. 遠程 Skill 不經過 `processPromptSlashCommand`——無 `!command` 替換、無 `$ARGUMENTS` 展開

## 完整生命週期總結

```
磁盤 SKILL.md
  ↓ parseFrontmatter()
  ↓ parseSkillFrontmatterFields() → 17 個字段
  ↓ createSkillCommand() → Command 對象
  ↓ 去重（realpath + seenFileIds）
  ↓ 條件 Skill → conditionalSkills Map（等待路徑匹配激活）
  ↓ getSkillDirCommands() memoize 快取
  ↓ getAllCommands() 合併 local + MCP
  ↓ formatCommandsWithinBudget() → 截斷後的 Skill 列表注入 System Prompt
  ↓ AI 選擇匹配的 Skill
  ↓ SkillTool.validateInput() → 名稱校驗 + 存在性檢查
  ↓ SkillTool.checkPermissions() → 四層權限檢查
  ↓ SkillTool.call() → inline 或 fork 執行
  ↓ contextModifier() → 注入 allowedTools + model + effort
  ↓ recordSkillUsage() → 更新使用頻率排名
```
