## 上下文窗口：200K 不是全部

Claude Code 的預設上下文窗口爲 200K tokens（`MODEL_CONTEXT_WINDOW_DEFAULT = 200_000`），但實際可用於對話的空間遠小於此：

```
上下文窗口（200K）
├── 系統提示詞（~15-25K，快取後成本低）
├── 工具定義（~10-20K，含 MCP 工具）
├── 用戶上下文（CLAUDE.md、git status 等）
├── 輸出預留（maxOutputTokens）
│   ├── 預設上限：64K
│   ├── 實際預設：8K（slot-reservation 優化）
│   └── 觸頂自動升級：一次 64K 重試
└── 剩餘：對話歷史空間（隨對話增長）
```

`getContextWindowForModel()`（`src/utils/context.ts:51`）按 5 級優先級解析窗口大小：

1. `CLAUDE_CODE_MAX_CONTEXT_TOKENS` 環境變量覆蓋
2. 模型名含 `[1m]` 後綴 → 1M tokens
3. `getModelCapability(model).max_input_tokens`
4. 1M beta header + 支援的模型（claude-sonnet-4, opus-4-6）
5. 兜底：200K

**有效上下文** = 窗口大小 - min(maxOutputTokens, 20K)，因爲壓縮摘要需要預留輸出空間。

## Token 計數：近似 vs 精確

系統使用兩級 token 計數策略：

### 近似估算（毫秒級）

```typescript
// src/services/tokenEstimation.ts
function roughTokenCountEstimation(content: string, bytesPerToken = 4): number {
  return Math.round(content.length / bytesPerToken)
}
```

對不同內容類型有特殊處理：
- **JSON/JSONL**：`bytesPerToken = 2`（密集的 `{`, `:`, `,` 符號，每個僅 1-2 token）
- **圖片/文件**：固定 2000 tokens（基於 2000×2000px 上限的保守估計）
- **thinking block**：按實際文本長度 / 4
- **tool_use**：序列化 `name + JSON.stringify(input)` 後 / 4

### 精確計數（API 呼叫）

使用 Anthropic 的 `beta.messages.countTokens` 端點。在不同 provider 上有不同路徑：

| Provider | 方法 |
|----------|------|
| Anthropic 直連 | `anthropic.beta.messages.countTokens()` |
| AWS Bedrock | `@aws-sdk/client-bedrock-runtime` 的 `CountTokensCommand` |
| Google Vertex | Anthropic SDK + beta 過濾 |
| 兜底（Bedrock 不支援） | 用 Haiku 發送 `max_tokens=1` 的請求，讀取 `usage.input_tokens` |

精確計數在關鍵決策點使用（壓縮前後對比、warning 判斷），近似估算在熱路徑使用（每輪循環的 shouldAutoCompact 檢查）。

## 自動壓縮的觸發閾值

```
src/services/compact/autoCompact.ts — 核心閾值
```

| 常量 | 值 | 含義 |
|------|----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 窗口減去此值 = 自動壓縮觸發點 |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | 在觸發點 + 20K 處顯示警告 |
| `ERROR_THRESHOLD_BUFFER_TOKENS` | 20,000 | 在觸發點 + 20K 處顯示錯誤 |
| `MANUAL_COMPACT_BUFFER_TOKENS` | 3,000 | 手動 /compact 的阻塞上限 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 連續失敗 3 次後停止嘗試 |

以 200K 窗口爲例：
- **~167K**：warning 閃爍，用戶看到建議壓縮的提示
- **~180K**：自動壓縮觸發（200K - 20K 輸出預留 = 180K 有效，再 - 13K buffer）
- **~197K**：達到 blocking limit，新訊息被阻止

`shouldAutoCompact()` 有多個逃逸條件：
- `compact` / `session_memory` 來源的查詢永不觸發（防遞歸死鎖）
- `DISABLE_COMPACT` / `DISABLE_AUTO_COMPACT` 環境變量
- 用戶設定 `autoCompactEnabled = false`
- Context Collapse 模式激活時抑制（collapse 自己管理上下文）
- Reactive Compact 實驗模式下抑制主動壓縮
- 超過連續失敗上限（circuit breaker）

## Micro-Compact：工具結果的漸進式壓縮

在觸發全量壓縮之前，系統先嚐試 **micro-compact**——只壓縮舊的工具呼叫結果：

```
可壓縮工具列表（COMPACTABLE_TOOLS）：
FileRead, Bash, Grep, Glob, WebSearch, WebFetch, FileEdit, FileWrite
```

策略基於時間：
- 超過一定時間（由 `timeBasedMCConfig` 控制）的工具結果被替換爲簡短佔位符
- 圖片/文件結果替換爲 `[image]` / `[document]` 文本
- 每次替換釋放 tokens，可能推遲全量壓縮

工具本身也有 `maxResultSizeChars`（通常 100K）硬限制，超長結果在寫入訊息前就被截斷。

## 全量壓縮的完整流程

```
autoCompactIfNeeded() / compactConversation()
  ↓
1. 執行 PreCompact hooks（外部可注入自定義指令）
  ↓
2. 嘗試 Session Memory 壓縮（更輕量，優先嚐試）
  ↓
3. Session Memory 失敗 → 全量壓縮
   a. 圖片/文件從訊息中剝離（替換爲 [image]/[document]）
   b. skill_discovery/skill_listing 附件剝離（壓縮後會重新注入）
   c. 通過 forked agent 發送摘要請求（複用主執行緒的 prompt cache）
   d. 如果摘要請求本身觸發 prompt-too-long → truncateHeadForPTLRetry()
      從最老的 API 輪次開始刪除，重試最多 3 次
   ↓
4. 壓縮成功後重建上下文：
   - compactBoundaryMarker（記錄壓縮類型、前 token 數等）
   - 摘要訊息（不可見的 user 訊息）
   - 最近 5 個文件的重新讀取（POST_COMPACT_TOKEN_BUDGET = 50K）
   - plan 文件附件（如果有）
   - plan mode 指令（如果在計劃模式中）
   - 已呼叫的 skill 內容（每 skill ≤5K，總計 ≤25K）
   - deferred tools / agent listing / MCP 指令的增量重新注入
   - SessionStart hooks 重新執行
   - PostCompact hooks 執行
  ↓
5. 更新快取基線，防止被誤判爲 cache break
```

### Prompt Cache Sharing

壓縮 API 呼叫是整個會話中最昂貴的操作之一。系統通過 `runForkedAgent` 複用主執行緒的快取前綴（system prompt + tools + context messages），將快取命中率從 2% 提升到接近 100%。這個優化單獨節省了艦隊級約 0.76% 的 `cache_creation` tokens。

## 輸出 Token 的 Slot 優化

一個經常被忽視的優化：**maxOutputTokens 的動態調整**。

```typescript
// src/services/api/claude.ts — getMaxOutputTokensForModel()
const defaultTokens = isMaxTokensCapEnabled()
  ? Math.min(maxOutputTokens.default, 8_000)  // 預設降到 8K
  : maxOutputTokens.default                     // 原始預設 32K/64K
```

爲什麼？因爲 API 的 slot 機制按 `max_tokens` 預留推理容量。BQ p99 輸出僅 4,911 tokens，32K 預設值浪費了 8-16 倍的 slot 容量。降到 8K 後，不到 1% 的請求被截斷——這些請求會自動獲得一次 64K 的 clean retry。

這個優化對 token 預算的影響是間接的：更多的 slot 容量意味着更少的排隊延遲，間接減少了超時和重試。

## Partial Compact：選擇性地壓縮

除了全量壓縮，用戶還可以在訊息歷史中選擇某個位置，只壓縮該位置之前或之後的內容：

- **`up_to` 方向**：壓縮選中訊息之前的內容，保留最近的對話
- **`from` 方向**：壓縮選中訊息之後的內容，保留早期的對話

`from` 方向保留 prompt cache（前綴不變），`up_to` 方向則破壞 cache（摘要插在保留內容之前）。

兩種方向的 PTL（prompt-too-long）重試策略相同：從最老的 API 輪次開始刪除，確保至少保留一組訊息供摘要。
