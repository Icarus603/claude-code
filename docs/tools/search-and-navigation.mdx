## 兩種搜索維度

| 維度 | 工具 | 底層實現 | 適用場景 |
|------|------|----------|---------|
| **按名稱找文件** | Glob | ripgrep `--files` + glob 過濾 | "找到所有測試文件"、"找 config 開頭的文件" |
| **按內容找程式碼** | Grep | ripgrep 正則搜索 | "哪裏定義了這個函數"、"誰在呼叫這個 API" |

兩者共享同一個 ripgrep 引擎，通過不同的參數組合實現不同搜索模式。

## ripgrep 的內嵌方式

Claude Code 不依賴系統安裝的 ripgrep——它在 `src/utils/ripgrep.ts` 中實現了三級降級策略：

```
優先級 1: 系統 ripgrep (USE_BUILTIN_RIPGREP=false)
  → 使用 PATH 中的 rg 二進制
  → 安全考慮：只用命令名 'rg'，不用完整路徑，防止 PATH 劫持

優先級 2: 內嵌模式 (bundled/native build)
  → process.execPath 自身，argv0='rg'
  → Bun 將 rg 靜態編譯進二進制，通過 argv0 分發

優先級 3: vendor 目錄 (npm build)
  → vendor/ripgrep/{arch}-{platform}/rg
  → macOS 需要 codesign 簽名 + 移除 quarantine xattr
```

平臺適配示例：
```
vendor/ripgrep/
  ├── x86_64-darwin/rg      # macOS Intel
  ├── arm64-darwin/rg        # macOS Apple Silicon
  ├── x86_64-linux/rg        # Linux Intel
  ├── arm64-linux/rg         # Linux ARM
  └── x86_64-win32/rg.exe    # Windows
```

### macOS 程式碼簽名

vendor 模式下的 rg 二進制需要 ad-hoc 簽名才能通過 Gatekeeper（`codesignRipgrepIfNecessary()`）：

```typescript
// 首次使用時執行：
// 1. 檢查是否已是有效簽名
codesign -vv -d <rg-path>
// 2. 如果只是 linker-signed，重新簽名
codesign --sign - --force --preserve-metadata=entitlements,requirements,flags,runtime <rg-path>
// 3. 移除隔離屬性
xattr -d com.apple.quarantine <rg-path>
```

## 搜索結果的設計考量

### head_limit 與 Token 預算

大型專案的搜索結果可能有數十萬條。預設最多返回 250 條匹配——這不是隨意選擇，而是**token 預算**的約束：

- 每條匹配行約 50-100 token
- 250 條 ≈ 12,500-25,000 token
- 這大約佔 200k 上下文窗口的 6-12%
- 超過這個比例，AI 的推理質量會下降

Grep 工具的 `head_limit` 參數讓 AI 可以按需調整——搜索小專案時可以用更大的值。

### 按修改時間排序

Glob 預設把**最近修改的文件排在前面**。這不是預設的檔案系統排序，而是刻意的設計決策：

```
設計假設：最近修改的文件最可能與當前任務相關
實際效果：AI 優先看到"活"的程式碼，而不是沉寂的歷史文件
```

在 `src/tools/GlobTool/` 中，ripgrep 的輸出在返回給 AI 前按 mtime 排序。

### ripgrep 的錯誤處理

ripgrep 執行有專門的錯誤恢復鏈（`src/utils/ripgrep.ts`）：

| 錯誤 | 處理 |
|------|------|
| **EAGAIN**（資源不足） | 自動以單執行緒模式 `-j 1` 重試 |
| **超時**（預設 20s，WSL 60s） | 返回已有部分結果，丟棄可能不完整的最後一行 |
| **緩衝區溢出** | 截斷到 20MB，返回已收集的結果 |
| **SIGTERM 失效** | 5 秒後升級爲 SIGKILL |

## ToolSearch：在 50+ 工具中發現目標

當可用工具超過 50 個時（含 MCP 提供的外部工具），AI 可能不知道該用哪個。**ToolSearch**（`src/tools/ToolSearchTool/`）提供了工具發現機制。

### 搜索算法

ToolSearch 實現了基於關鍵詞的加權搜索（`searchToolsWithKeywords()`）：

```
輸入: query = "database connection"
     ↓
1. 精確匹配: 檢查是否有工具名完全匹配（快速路徑）
2. MCP 前綴匹配: "mcp__postgres" → 匹配所有 postgres 相關工具
3. 關鍵詞拆分: ["database", "connection"]
4. 工具名解析:
   - MCP 工具: "mcp__server__action" → ["server", "action"]
   - 普通工具: "FileEditTool" → ["file", "edit", "tool"]
5. 加權評分:
   - 工具名精確匹配: 10 分（MCP: 12 分）
   - 工具名部分匹配: 5 分（MCP: 6 分）
   - searchHint 匹配: 4 分
   - 描述匹配: 2 分
6. 必選詞過濾: "+database" 前綴表示必須包含
7. 按分數排序，返回 top-N
```

### `select:` 直接選擇

AI 也可以用 `select:ToolName` 精確選擇已知工具。這比搜索更快，且支援逗號分隔的批量選擇（`select:A,B,C`）。

### 延遲加載（Deferred Tools）

不是所有工具都常駐內存。MCP 工具和低頻工具被標記爲 `isDeferredTool`，只有在 ToolSearch 選中後才真正加載。這減少了每次 API 呼叫的 token 開銷（工具描述佔用大量 token）。

### 快取策略

工具描述的取得是 memoized 的——只在延遲工具集合變化時清除快取：

```typescript
// 工具名排序後拼接作爲快取 key
function getDeferredToolsCacheKey(deferredTools: Tools): string {
  return deferredTools.map(t => t.name).sort().join(',')
}
```

## Web 搜索與抓取

AI 的信息取得不侷限於本地程式碼：

- **WebSearch**（`src/tools/WebSearchTool/`）：呼叫 Anthropic API 的 `web_search_20250305` server tool 搜索互聯網
- **WebFetch**（`src/tools/WebFetchTool/`）：抓取特定 URL 內容，轉換爲 Markdown 供 AI 閱讀

這讓 AI 可以查閱文件、搜索 Stack Overflow、閱讀 GitHub issue——和人類開發者的工作方式一致。

### WebSearch 實現機制

WebSearch 通過適配器模式支援兩種搜索後端，由 `src/tools/WebSearchTool/adapters/` 中的工廠函數 `createAdapter()` 選擇：

```
適配器架構:
  WebSearchTool.call()
    → createAdapter() 選擇後端
      ├─ ApiSearchAdapter — Anthropic API 服務端搜索（需官方 API 密鑰）
      └─ BingSearchAdapter — 直接抓取 Bing 搜索頁面解析（無需 API 密鑰）
    → adapter.search(query, options)
    → 轉換爲統一 SearchResult[] 格式返回
```

#### 適配器選擇邏輯

`adapters/index.ts` 中的工廠函數按以下優先級選擇後端：

| 優先級 | 條件 | 適配器 |
|--------|------|--------|
| 1 | 環境變量 `WEB_SEARCH_ADAPTER=api` | `ApiSearchAdapter` |
| 2 | 環境變量 `WEB_SEARCH_ADAPTER=bing` | `BingSearchAdapter` |
| 3 | API Base URL 指向 Anthropic 官方 | `ApiSearchAdapter` |
| 4 | 第三方代理 / 非官方端點 | `BingSearchAdapter` |

適配器是無狀態的，同一會話內快取複用。

#### ApiSearchAdapter — API 服務端搜索

將搜索請求委託給 Anthropic API 的 `web_search_20250305` server tool：

```
呼叫鏈:
  ApiSearchAdapter.search(query, options)
    → queryModelWithStreaming() 發起獨立的 API 呼叫
      → 攜帶 extraToolSchemas: [BetaWebSearchTool20250305]
      → API 服務端執行搜索，返回流式事件
        → server_tool_use / web_search_tool_result / text 交替返回
    → extractSearchResults() 從 content blocks 提取 SearchResult[]
```

| 特性 | 實現 |
|------|------|
| **模型選擇** | Feature flag `tengu_plum_vx3` 控制用 Haiku（強制 tool_choice）還是主模型 |
| **搜索上限** | 每次呼叫最多 8 次搜索（`max_uses: 8`） |
| **域過濾** | 支援 `allowedDomains` / `blockedDomains` |
| **進度追蹤** | 流式解析 `input_json_delta` 提取 query，實時回調 `onProgress` |

#### BingSearchAdapter — Bing 搜索頁面解析

直接抓取 Bing 搜索 HTML 並用正則提取結果，無需 API 密鑰：

```
呼叫鏈:
  BingSearchAdapter.search(query, options)
    → axios.get(bing.com/search?q=...)  — 使用瀏覽器級別 headers 繞過反爬
    → extractBingResults(html)
      → 正則匹配 <li class="b_algo"> 塊
      → 提取 <h2><a> 標題和 URL
      → resolveBingUrl() 解碼 Bing 重定向連結
      → extractSnippet() 三級降級提取摘要
    → 客戶端域過濾 (allowedDomains / blockedDomains)
    → 返回 SearchResult[]
```

**反爬策略**：Bing 對非瀏覽器 UA 返回需要 JS 渲染的空頁面。適配器使用完整的 Edge 瀏覽器請求頭（包含 `Sec-Ch-Ua`、`Sec-Fetch-*` 等現代瀏覽器標頭）確保獲得完整 HTML。同時使用 `setmkt=en-US` 參數統一市場定位，避免 Bing 基於用戶 IP 做區域化定向（如跳轉到德語/新加坡市場導致結果不相關）。

**URL 解碼**：Bing 搜索結果中的 URL 爲重定向格式（`bing.com/ck/a?...&u=a1aHR0cHM6Ly9...`），`resolveBingUrl()` 從 `u` 參數中 base64 解碼出真實目標 URL（`a1` 前綴 = https，`a0` = http）。

**摘要提取**（`extractSnippet()`）按優先級嘗試三個來源：
1. `<p class="b_lineclamp...">` — 帶行截斷的摘要段落
2. `<div class="b_caption">` 內的 `<p>` — 普通摘要段落
3. `<div class="b_caption">` 的直接文本內容 — 兜底方案

| 特性 | 實現 |
|------|------|
| **超時** | 30 秒（`FETCH_TIMEOUT_MS`） |
| **域過濾** | 支援 `allowedDomains` / `blockedDomains`，含子域名匹配 |
| **進度追蹤** | 發送 query_update 和 search_results_received 回調 |
| **中止支援** | 外部 AbortSignal 傳播到 axios 請求 |

### WebSearchTool 統一介面

`WebSearchTool`（`src/tools/WebSearchTool/WebSearchTool.ts`）是面向主循環的工具定義，所有 provider 均可使用（`isEnabled()` 始終返回 true）。它將適配器返回的 `SearchResult[]` 轉換爲內部 `Output` 格式，`mapToolResultToToolResultBlockParam` 將搜索結果格式化爲帶 markdown 超連結的文本，並附加 "REMINDER" 要求主模型在回覆中包含 Sources。

### WebFetch 實現機制

WebFetch 是一個完整的 HTTP 客戶端 + 內容處理管線：

```
呼叫鏈:
  WebFetchTool.call({ url, prompt })
    → getURLMarkdownContent(url)
      → validateURL() — 長度≤2000、無用戶名密碼、公網域名
      → URL_CACHE 命中檢查（15 分鐘 TTL LRU，50MB 上限）
      → checkDomainBlocklist() — 呼叫 api.anthropic.com/api/web/domain_info 預檢
      → getWithPermittedRedirects() — axios 請求，自定義重定向處理
        → HTML → Turndown 轉 Markdown（懶加載單例，~1.4MB）
        → 非 HTML → 原始文本
        → 二進制（PDF 等）→ persistBinaryContent() 保存到磁盤
    → applyPromptToMarkdown()
      → 截斷到 100K 字符
      → queryHaiku() 用小模型按 prompt 提取信息
    → 返回處理後的結果
```

安全防護多層設計：

| 層級 | 機制 | 說明 |
|------|------|------|
| **域名預檢** | `checkDomainBlocklist()` | 呼叫 `api.anthropic.com/api/web/domain_info?domain=…`，5 分鐘快取 |
| **重定向控制** | `isPermittedRedirect()` | 僅允許同 host（±www）重定向，跨域重定向返回提示讓 AI 重新呼叫 |
| **重定向深度** | `MAX_REDIRECTS = 10` | 防止重定向循環無限掛起 |
| **內容大小** | `MAX_HTTP_CONTENT_LENGTH = 10MB` | 單次響應上限 |
| **請求超時** | `FETCH_TIMEOUT_MS = 60s` | 主請求超時；域名預檢 10s |
| **URL 驗證** | `validateURL()` | 長度、協議、用戶名密碼、公網域名檢查 |
| **egress 檢測** | `X-Proxy-Error: blocked-by-allowlist` | 檢測企業代理攔截 |

預批准域名（`src/tools/WebFetchTool/preapproved.ts`）：

用戶無需手動授權即可抓取的域名列表，包含 ~90 個主流技術文件站點（MDN、Python docs、React docs、AWS docs 等）。列表分爲 hostname-only 和 path-prefix 兩類，查找複雜度 O(1)。

對預批准域名，WebFetch 跳過 Haiku 摘要步驟（如果內容是 Markdown 且 < 100K 字符），直接返回原文——因爲技術文件本身的結構化程度已經足夠好。

權限模型方面，WebFetch 按 hostname 生成 `domain:xxx` 規則匹配用戶的 allow/deny/ask 規則，支援用戶對特定域名設定永久允許或拒絕。

### ripgrep 的流式輸出

對於交互式場景（如 QuickOpen），ripgrep 支援**流式輸出**（`ripGrepStream()`）：

```
rg --files → 逐 chunk 到達 → 按行分割 → onLines(lines) 回調
```

不需要等 ripgrep 完成整個搜索——第一批結果在 rg 仍在遍歷目錄樹時就已展示。呼叫者可以通過 AbortSignal 提前終止搜索（例如找到足夠多的結果後）。
