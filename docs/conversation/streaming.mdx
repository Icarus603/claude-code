## 爲什麼需要流式

想象 AI 需要 30 秒才能生成完整回答——如果等 30 秒後才一次性顯示，用戶體驗是災難性的。

流式響應讓用戶**實時看到 AI 的思考過程**：
- 文字逐字出現，用戶能提前判斷方向是否正確
- 工具呼叫的參數在生成過程中就能預覽
- 長時間任務不會讓用戶覺得"卡死了"

## `BetaRawMessageStreamEvent` 核心事件類型

流式 API 返回的是一系列 `BetaRawMessageStreamEvent`，每種事件類型對應流式響應的不同階段（`src/services/api/claude.ts`）：

```
message_start           ← 訊息開始，包含 model、usage 初始值
  ├── content_block_start   ← 內容塊開始（text / tool_use / thinking）
  │   ├── content_block_delta  ← 增量資料（text_delta / input_json_delta / thinking_delta）
  │   ├── content_block_delta  ← ... 持續到達
  │   └── content_block_stop   ← 內容塊結束，yield AssistantMessage
  ├── content_block_start   ← 下一個內容塊...
  │   └── ...
  └── message_delta       ← stop_reason + 最終 usage
message_stop            ← 訊息結束
```

### 事件處理狀態機

`src/services/api/claude.ts:1980-2298` 實現了一個基於 `switch(part.type)` 的狀態機：

| 事件類型 | 處理邏輯 | 狀態變更 |
|----------|----------|----------|
| `message_start` | 初始化 `partialMessage`，記錄 TTFT（首字節延遲） | `usage` 初始化 |
| `content_block_start` | 按 `part.index` 建立對應類型的內容塊 | `contentBlocks[index]` 初始化 |
| `content_block_delta` | 按子類型增量追加資料 | text / thinking / input 累加 |
| `content_block_stop` | 構建完整 `AssistantMessage` 並 yield | 訊息推入 `newMessages` |
| `message_delta` | 更新 stop_reason 和最終 usage | 寫回最後一條訊息 |
| `message_stop` | 無操作（流結束標記） | — |

### 內容塊類型及其增量資料

`content_block_start` 中的 `content_block.type` 決定了如何處理後續 delta：

| 內容塊類型 | Delta 類型 | 累加邏輯 |
|-----------|-----------|----------|
| `text` | `text_delta` | `text += delta.text` |
| `thinking` | `thinking_delta` + `signature_delta` | `thinking += delta.thinking`，`signature = delta.signature` |
| `tool_use` | `input_json_delta` | `input += delta.partial_json`（JSON 字符串增量拼接） |
| `server_tool_use` | `input_json_delta` | 同 tool_use |
| `connector_text` | `connector_text_delta` | 特殊連接器文本（feature flag 控制） |

關鍵設計：`content_block_start` 時所有文本字段初始化爲空字符串，只通過 `content_block_delta` 累加。這是因爲 SDK 有時在 start 和 delta 中重複發送相同文本。

## 文本 chunk 和 tool_use block 的交織

一次 AI 響應可能包含多個內容塊，交替出現：

```
content_block_start (text, index=0)     "我來幫你修復這個 bug。"
content_block_delta  (text_delta)       "首先..."
content_block_stop  (index=0)
content_block_start (tool_use, index=1) { name: "Read", input: "..." }
content_block_delta  (input_json_delta) '{"file_p' → 'ath":' → '"src/foo.ts"}'
content_block_stop  (index=1)
content_block_start (text, index=2)     "我已經看到了問題所在..."
content_block_stop  (index=2)
```

每個 `content_block_stop` 觸發一次 `yield`，將完整的 AssistantMessage 推送給消費者。這意味着一個 AI 響應會產生**多條** `AssistantMessage`——文本訊息和工具呼叫訊息交替產出。

`stop_reason` 要等到 `message_delta` 才確定（可能是 `end_turn`、`tool_use`、`max_tokens` 等），所以最後一條訊息的 `stop_reason` 是**回寫**的：

```typescript
// claude.ts:2246 — 直接屬性修改，不用對象替換
// 因爲 transcript 寫隊列持有 message.message 的引用
const lastMsg = newMessages.at(-1)
if (lastMsg) {
  lastMsg.message.usage = usage
  lastMsg.message.stop_reason = stopReason
}
```

## 流式中的錯誤處理

### 網絡斷開

流式連接依賴 SSE（Server-Sent Events）。當連接中斷時：

1. **Stream idle watchdog**：定時檢測事件間隔，超過閾值（stall）觸發告警和重試
2. **Stream abort**：如果 watchdog 檢測到長時間無事件，拋出錯誤進入重試流程
3. **非流式降級**：作爲最後手段，回退到非流式請求（一次性取得完整響應）

```typescript
// claude.ts:2338-2355 — 檢測空流
// 1. 完全沒有事件 → 代理返回了非 SSE 響應
// 2. 有 message_start 但沒有 content_block_stop → 流被截斷
```

### API 限流

當 API 返回限流錯誤時，系統使用 `withRetry` 包裝器進行指數退避重試。重試邏輯考慮了：
- 錯誤類型（429 限流 vs 500 服務器錯誤）
- 重試次數上限
- 退避間隔

### Token 超限

兩種 token 超限場景有不同的處理：

| 場景 | stop_reason | 處理方式 |
|------|------------|----------|
| **輸出超限** | `max_tokens` | 生成錯誤訊息，建議設置 `CLAUDE_CODE_MAX_OUTPUT_TOKENS` |
| **上下文窗口超限** | `model_context_window_exceeded` | 觸發 compaction 壓縮對話歷史後重試 |

```typescript
// claude.ts:2267-2293
if (stopReason === 'max_tokens') {
  yield createAssistantAPIErrorMessage({ error: 'max_output_tokens', ... })
}
if (stopReason === 'model_context_window_exceeded') {
  // 複用 max_output_tokens 的恢復路徑
  yield createAssistantAPIErrorMessage({ error: 'max_output_tokens', ... })
}
```

### 流式停滯檢測

系統持續監控事件到達間隔，檢測"停滯"（stall）：

```typescript
// claude.ts:1940-1966
const STALL_THRESHOLD_MS = 10_000  // 10 秒無事件視爲停滯
if (timeSinceLastEvent > STALL_THRESHOLD_MS) {
  stallCount++
  totalStallTime += timeSinceLastEvent
  logEvent('tengu_streaming_stall', { stall_duration_ms, stall_count, ... })
}
```

多個 stall 累積後，watchdog 可能決定中斷流並觸發重試。

## 工具執行的流式反饋

BashTool 的命令執行也是流式的——通過 `onProgress` 回調逐行推送輸出：

```
BashTool.call() → runShellCommand() → AsyncGenerator
  ├── 每秒輪詢輸出文件 → onProgress(lastLines, allLines, ...)
  ├── yield { type: 'progress', output, fullOutput, elapsedTimeSeconds }
  └── return { code, stdout, interrupted, ... }
```

UI 層通過 `useToolCallProgress` hook 實時展示命令輸出，而不是等命令完全結束。長時間執行的命令還支援自動後臺化（`shouldAutoBackground`）。

## 多 Provider 適配

| Provider | 流式協議 | 特殊處理 |
|----------|----------|----------|
| **Anthropic Direct** | 原生 SSE | 延遲最低，TTFT 最快 |
| **AWS Bedrock** | AWS SDK 流式介面 | 需要額外的 beta header 和認證 |
| **Google Vertex** | gRPC → 事件流 | 通過 `getMergedBetas()` 適配 |
| **Azure** | Anthropic 相容 API | 自定義 base URL |

所有 Provider 通過統一的 `Stream<BetaRawMessageStreamEvent>` 抽象層屏蔽差異。上層程式碼（QueryEngine、REPL）不需要關心底層用的是哪個 Provider。

### Provider 選擇

`src/utils/model/providers.ts` 中的 `getAPIProvider()` 根據設定決定使用哪個 Provider：

```typescript
// 根據 api_provider 設定選擇：
// "anthropic" → 直連
// "bedrock"   → AWS SDK
// "vertex"    → Google SDK
// 第三方 base URL → 自動檢測
```

每個 Provider 需要適配的細節包括：認證方式、beta header、請求參數格式、錯誤碼映射——但這些差異在 `claude.ts` 的 `queryStream()` 函數中被統一處理。
