## feature() 是什麼

Claude Code 使用 Bun 打包器的 `bun:bundle` 模組提供編譯時特性門控：

```typescript
// 源碼中的用法（src/tools.ts 等）
import { feature } from 'bun:bundle'

const SleepTool = feature('PROACTIVE') || feature('KAIROS')
  ? require('./tools/SleepTool/SleepTool.js').SleepTool
  : null
```

在 Anthropic 的內部構建中，`feature()` 在打包時被求值——返回 `true` 的程式碼會被保留，返回 `false` 的程式碼會被 **Dead Code Elimination (DCE)** 徹底移除。

在我們的反編譯版本中，這個函數被兜底爲：

```typescript
// src/entrypoints/cli.tsx 第 3 行
const feature = (_name: string) => false;
```

這意味着所有 88+ 個 feature flag 後的程式碼**在執行時永遠不會執行**，但程式碼本身完整保留，可以閱讀和分析。

## 程式碼中的典型模式

Feature flags 在程式碼中主要有三種使用模式：

### 模式一：條件加載工具

```typescript
// src/tools.ts — 最常見的模式
const MonitorTool = feature('MONITOR_TOOL')
  ? require('./tools/MonitorTool/MonitorTool.js').MonitorTool
  : null
```

當 flag 爲 `false` 時，`require()` 呼叫被 DCE 移除，工具不會出現在可用工具列表中。

### 模式二：條件註冊命令

```typescript
// src/commands.ts — 註冊斜槓命令
if (feature('VOICE_MODE')) {
  commands.push({ name: 'voice', description: '...' })
}
```

### 模式三：條件啓用 API 特性

```typescript
// src/constants/betas.ts — 控制發送給 API 的 beta header
export const AFK_MODE_BETA_HEADER = feature('TRANSCRIPT_CLASSIFIER')
  ? 'afk-mode-2026-01-31'
  : ''
```

<Note>
由於 `feature()` 在構建時求值，被 DCE 移除的程式碼不會增加最終打包體積。但在反編譯版本中，這些程式碼全部保留——這正是我們能夠進行完整分析的原因。
</Note>

## 有趣的發現

- **KAIROS 家族**最龐大——6 個相關 flag 控制從核心功能到推送通知的方方面面
- **ABLATION_BASELINE** 是用於"科學對照實驗"的——它會關閉 thinking、compaction、auto-memory 等高級功能，測量裸 API 呼叫的基線性能
- **ULTRAPLAN** 和 **ULTRATHINK** 暗示着比當前 extended thinking 更高級的推理模式
